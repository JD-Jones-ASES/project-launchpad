AP CSP — Topic 1.5: Data Compression: Lossless vs. Lossy and Metadata

Core Question

A raw 1920x1080 photograph requires about 6 MB. Three minutes of stereo CD-quality audio takes about 31 MB. Uncompressed HD video runs at roughly 150 MB per second. Without compression, the internet as it currently exists would be impossible: streaming video would saturate every backbone within seconds, and a 256 GB phone would hold fewer than 500 photos. Data compression is not an optimization — it is a foundational necessity. This section examines how compression works, the crucial distinction between lossless (every bit recovered) and lossy (perceptually-driven loss) approaches, and the role of metadata in describing the files that result. Both layers — primary content and metadata — sit on top of the bit-level representation introduced in earlier sections.

Key Concepts

  • Compression: Reducing the bits needed to represent information by exploiting redundancy. Most real data has patterns — repeated values, predictable structure — that allow shorter encodings.
  • Compression ratio: Original size divided by compressed size. A 4:1 ratio means the compressed file is one-quarter the original.
  • Lossless compression: Reproduces the EXACT original file on decompression. Every bit identical. Compression ratios typically 1.5:1 to 4:1 for images.
  • Lossy compression: Discards information judged to be perceptually unimportant. The original cannot be recovered. Compression ratios typically 10:1 to 20:1 for images.
  • Run-length encoding (RLE): The simplest lossless technique. Replaces consecutive identical values with a count + value pair. Great for solid-color regions; weak for photographs.
  • DEFLATE: The lossless algorithm inside ZIP, PNG, and gzip. Combines Huffman coding (short codes for common symbols, long codes for rare symbols) with LZ77 dictionary compression (back-references to earlier byte sequences).
  • PNG: Lossless image format using DEFLATE. Used for screenshots, logos, icons, medical imaging.
  • JPEG: Lossy image format using the discrete cosine transform (DCT) on 8x8 blocks, then quantization of high-frequency coefficients. The human visual system is less sensitive to high-frequency content, so aggressive quantization is usually imperceptible.
  • MP3 / AAC / Opus: Lossy audio codecs using a psychoacoustic model. Discard sounds humans cannot perceive (masked by louder sounds, above audible range, very quiet).
  • MP4 / H.264: Lossy video codecs. Store reference (key) frames at intervals; encode other frames as motion-compensated differences from neighbors. Exploits temporal redundancy.
  • FLAC: Lossless audio at ~2:1 compression. Preferred for archival masters and audiophile distribution.
  • Metadata: Information about a file (creation date, author, GPS coords, camera model). Not part of the primary content but travels with it. Stored in the file header (EXIF for JPEG, ID3 for MP3) or a sidecar file (.xmp for Lightroom).
  • EXIF: Metadata standard for JPEG, including camera model, exposure settings, GPS coordinates, timestamp. Privacy implication: GPS reveals where the photo was taken.

Worked Examples / Code Walkthroughs

Example 1 — RLE compression on a row of pixels:

Original: white white white white white black black white white   (9 values)
RLE:      (5, white) (2, black) (2, white)                          (3 groups)
Ratio:    9 / 3 = 3:1

For an image with many large, solid-color regions (a logo, a screenshot of a UI), RLE achieves high ratios. For a photograph (where every pixel is a slightly different color), RLE may even increase the file size because adding count metadata to runs of length 1 wastes bits.

Example 2 — Compression ratio arithmetic:

A ZIP archive: original 50 MB, ratio 5:1
compressed = original / ratio = 50 / 5 = 10 MB

A JPEG: original raw 72 MB, JPEG 8 MB
ratio = 72 / 8 = 9:1

A PNG: original 40 MB, PNG 12 MB
ratio = 40 / 12 ≈ 3.3:1

Example 3 — JPEG re-encoding degradation:

from PIL import Image
img = Image.open('original.jpg')
for i in range(10):
    img.save(f'pass_{i}.jpg', 'JPEG', quality=80)
    img = Image.open(f'pass_{i}.jpg')
# Inspecting pass_0.jpg vs. pass_9.jpg shows progressive blocking,
# color banding, and edge artifacts -- each pass discards more detail.

The lesson: never save edits in JPEG. Edit in PNG (lossless) and export to JPEG only as the final delivery copy.

Example 4 — Professional photo workflow:

1. RAW capture          (20-50 MB)  -- camera sensor data + EXIF
2. Edit in Lightroom    -- adjustments saved to .xmp sidecar (small text file)
3. Web export           (~3 MB)     -- JPEG at quality 90, EXIF GPS stripped
4. Archival export      (~50 MB)    -- TIFF or PNG with all metadata preserved

At each step a deliberate choice is made between fidelity and file size, and metadata is augmented or filtered.

Common Pitfalls

  • Re-encoding JPEGs. Each save discards more high-frequency detail. After many passes, blocking and banding become visible. Edit in PNG; export to JPEG only at the final delivery step.
  • Confusing lossless with no compression. Lossless still compresses (1.5:1 to 4:1 typical); the guarantee is that decompression returns the EXACT original, not that no compression happens.
  • Believing compression always shrinks files. Already-compressed data (JPEGs, MP3s, ZIPs) cannot be further compressed losslessly. Trying may even increase the file size by adding overhead.
  • Sharing photos with EXIF intact. GPS coordinates can reveal home address, school, or daily commute. Strip EXIF before public sharing, and rely on social network EXIF-stripping for upload but not for direct file transfers.
  • Using lossy compression for medical or archival data. Diagnostic imaging cannot tolerate any pixel-level loss; archival masters need bit-exact preservation. Use PNG/TIFF/DICOM-lossless or FLAC for these contexts.

Connect to the Exam

Topic 1.5 anchors Big Idea 2 (Data) and is a frequent source of stimulus-style exam questions. Expect: (1) a compression-ratio arithmetic problem (given original and ratio, find compressed; or vice versa), (2) a question asking to identify lossless vs. lossy from format names (PNG/ZIP/FLAC = lossless; JPEG/MP3/MP4 = lossy), (3) a real-world tradeoff scenario (medical imaging vs. social sharing) and asking which compression strategy fits, and (4) a privacy question about metadata. The exam will not require you to implement DCT or Huffman coding, but you should know that JPEG uses DCT, and that lossless compression is bounded by Shannon entropy.

Self-Check

  • A graphic designer re-saves the same JPEG image 10 times in a row. Explain what happens to image quality and why.
  • A ZIP archive containing text files achieves a 5:1 compression ratio. If the original files total 50 MB, how large is the ZIP file? Conversely, if the ZIP file is 8 MB, how large is the original?
  • Give one specific reason why sharing a smartphone photo of your home on a public website without stripping metadata could be a privacy risk.
  • Why would re-zipping a ZIP file likely make the result LARGER than the input, not smaller? (Hint: think about Shannon entropy and the residual redundancy in compressed output.)
  • A music streaming service offers two tiers: 256 kb/s MP3 and FLAC at ~1,000 kb/s. Which tier consumes more bandwidth, and which tier matters more to a casual listener vs. an audiophile? Justify both answers with specific properties of MP3 and FLAC.
