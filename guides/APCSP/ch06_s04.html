AP CSP — Topic 6.4: Theoretical Computing Limits: Undecidable Problems

Core Question

Why are some problems not just slow but provably impossible for any algorithm to solve, and how does Turing's diagonalization proof of the Halting Problem establish that boundary?

Key Concepts

  • A problem is decidable if some algorithm correctly answers YES or NO in finite time for every possible input. Sorting, searching, and primality testing are decidable.
  • A problem is undecidable if no algorithm can correctly answer YES or NO for every possible input. Any proposed algorithm must either give a wrong answer somewhere or fail to halt.
  • Undecidability is a proven mathematical impossibility, not a "we haven't found it yet" situation. It applies to every model of computation, including future hardware.
  • The canonical undecidable problem is the Halting Problem: given an arbitrary program P and an arbitrary input I, decide whether P halts when run on I.
  • Intractable (Section 6.3) means correctly solvable but requires unreasonable time. Undecidable means no correct algorithm exists at all. Different categories.
  • Turing's 1936 proof uses diagonalization. Assume halts(program, input) exists, then construct a procedure diagonal that does the OPPOSITE of whatever halts predicts. Running diagonal(diagonal) contradicts halts in both possible cases.
  • The proof produces consequences: no general-purpose bug detector for infinite loops, and no general-purpose program verifier that proves arbitrary specs. Specific cases remain solvable.
  • Detecting SOME infinite loops is still practical. The result only forbids a single general algorithm that works for every program-input pair.
  • Universal Turing machines can simulate any other Turing machine. Modern general-purpose computers are universal in this sense, so undecidability transfers to every real-world computer.
  • Alan Turing (1912-1954) wrote the proof in his 1936 paper "On Computable Numbers." He later led WW2 codebreaking at Bletchley Park. He was honored posthumously on the British 50 banknote in 2021.

Worked Examples / Code Walkthroughs

Example 1 — The hypothetical halts procedure (which cannot exist)

PROCEDURE halts(program, input)
{
    // Would return TRUE if program halts on input,
    // FALSE if program loops forever.
    // PROOF: no such procedure can be written.
}

The proof does not depend on what halts looks like inside — only on the assumption that it works correctly for every program-input pair. The contradiction is reached without ever needing to implement halts.

Example 2 — The diagonal construction

PROCEDURE diagonal(P)
{
    IF (halts(P, P) == TRUE)
        REPEAT UNTIL (FALSE) {
            // do nothing — loop forever
        }
    ELSE
        RETURN     // halt immediately
}

Diagonal contradicts halts's prediction by design. If halts says "P halts on input P," diagonal does the opposite (loops forever). If halts says "P loops on input P," diagonal halts.

Example 3 — The crucial call: diagonal(diagonal)

When we feed diagonal to itself, halts(diagonal, diagonal) must return either True or False. Trace both cases:

  • Case A (halts returns True): halts is asserting that diagonal halts when run on diagonal. But by construction, when halts says "halts," diagonal enters the infinite loop — so diagonal does NOT halt. Contradiction.
  • Case B (halts returns False): halts is asserting that diagonal loops forever. But by construction, when halts says "loops," diagonal halts immediately. Contradiction.

Either branch of the if/else makes halts wrong. The only way out is that the original assumption (halts exists and works correctly) is false. Therefore no such procedure can exist.

Example 4 — Distinguishing intractable from undecidable

Problem Category Why
Sort a list of n integers Decidable + tractable Merge sort runs in O(n log n)
Traveling Salesman (exact) Decidable + intractable (n-1)! routes; correct but exponential
Halting Problem Undecidable Provably no general algorithm exists

The Halting Problem stands alone: no amount of compute time helps because there is no correct algorithm to run.

Common Pitfalls

  • Saying "undecidable means no one has found the answer yet." Undecidable means PROVEN impossible.
  • Confusing undecidable with intractable. Intractable problems can be solved correctly given exponential time; undecidable problems cannot be solved correctly at all.
  • Believing the proof requires checking infinite cases empirically. It is a logical proof by contradiction — no programs are actually tested.
  • Believing modern AI or quantum computers solve the Halting Problem. They do not. Mathematical limits apply to every Turing-complete model.
  • Concluding that no infinite loop can ever be detected. SPECIFIC loops are detected every day; the result only rules out a single GENERAL algorithm that works on all program-input pairs.

Connect to the Exam

The AP CED includes a topic on undecidable problems and theoretical limits. Multiple-choice questions test the intractable-vs-undecidable distinction, name the Halting Problem as the canonical example, and ask you to identify Turing's role and the diagonalization technique. Free-response questions can ask you to explain in plain language why no halting-detection algorithm can exist. The Create PT does not require this material directly, but the language of "any algorithm" appears in the Algorithm Implementation rubric row, and a clean grasp of computability boundaries makes you better at justifying procedure correctness.

Self-Check

  1. In one sentence, explain the difference between an intractable problem and an undecidable problem.
  2. State the Halting Problem in your own words. Why is it relevant to programmers, not just theorists?
  3. In two or three sentences, describe what the diagonal procedure does and why it leads to a contradiction.
  4. Does the undecidability of the Halting Problem mean programmers can never detect any infinite loops? Explain.
  5. What was Alan Turing's 1936 paper called, and what mathematical model did it introduce?
