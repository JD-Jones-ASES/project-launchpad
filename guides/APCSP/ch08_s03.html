AP CSP — Topic 8.3: Bias in Computing Systems — Mechanisms and Mitigations

Core Question

Why do machine-learning models often produce systematically less accurate or differently distributed outputs for some categories of input than for others, and what technical responses are available? Section 8.3 frames computing bias as a statistical phenomenon with three specific mechanisms — sampling bias, feature selection bias, and the evaluation gap — each with a corresponding technical mitigation. The framing is mechanism-focused throughout: the textbook's job is to let students explain how skewed data produces skewed models and how each mitigation targets a specific source of skew. The three mechanism-mitigation pairs form the AP-tested vocabulary.

Key Concepts

  • Computing bias: A systematic error in computing-system outputs, distinct from random error because the errors are correlated with input category. Bias can arise from training-data composition, feature selection, evaluation procedure, or biased historical data.
  • Training data: The labeled dataset used to fit a model's parameters. Its statistical properties — size, diversity, label quality, distribution across input types — directly determine model behavior.
  • Sampling bias: Mismatch between training-set composition and the deployed input distribution. Sampling bias is a data property that exists before training.
  • Loss-function intuition: Optimizers minimize average error across training examples. With 90/10 class imbalance, the 90% class dominates the optimization signal, producing systematically lower error there than in the 10% class.
  • Feature selection bias: Bias arising from chosen features that serve as proxies for protected attributes.
  • Proxy variable: A feature statistically correlated with an attribute the model is not supposed to use (ZIP code, first name, graduation year + birth date, commute distance), allowing the model to implicitly depend on the protected attribute.
  • Evaluation gap: A procedural failure to disaggregate performance metrics. Aggregate accuracy can hide large per-subgroup disparities (e.g., 99% majority vs 70% minority hidden under 95% aggregate).
  • Dataset rebalancing: Mitigation for sampling bias — oversample under-represented classes, undersample over-represented classes, or generate synthetic examples via data augmentation.
  • Feature auditing: Mitigation for feature selection bias — test candidate features for correlation with protected attributes pre-training; use SHAP or similar attribution methods post-training to detect proxy-variable usage.
  • Subgroup performance metrics and model cards: Mitigation for the evaluation gap. Compute accuracy/precision/recall/error rate separately per subgroup. A model card is a standardized documentation artifact recording intended use, training-data provenance, evaluation methodology, and per-subgroup metrics.

Mechanisms and Mitigations

  • Mechanism: Sampling bias -> Mitigation: Dataset rebalancing. A spam filter trained on 95% English email misclassifies legitimate Spanish messages because the under-represented class contributed too little optimization signal. Rebalancing the training set (oversampling Spanish examples, undersampling English, or synthesizing new Spanish examples) aligns the training distribution with the deployed distribution. The facial-recognition error-rate gap (5-35% on darker-skinned female faces vs under 2% on lighter-skinned male faces) is the textbook's worked example: under-represented demographic groups contributed fewer training examples, so the model optimized less thoroughly for their characteristics.
  • Mechanism: Feature selection bias via proxy variables -> Mitigation: Feature auditing. A loan-approval algorithm using ZIP code can produce systematically different outputs by ZIP code that mirror historical demographic patterns even when race was not provided as a feature. Mitigation involves measuring correlation between candidate features and protected attributes before training, removing or replacing high-correlation features, and using SHAP to detect proxy-variable usage in trained models.
  • Mechanism: Multiple correlated proxies -> Mitigation: Subgroup-level outcome evaluation. Removing one proxy does not always resolve the issue if other correlated proxies remain. Beyond feature removal, fairness constraints during training and evaluation at the subgroup level let practitioners detect whether bias persists through indirect channels.
  • Mechanism: Evaluation gap -> Mitigation: Disaggregated evaluation and model cards. Computing performance metrics stratified by relevant subgroups reveals hidden disparities. A model card records training-data provenance, evaluation methodology, and per-subgroup results, enabling downstream users to make informed deployment decisions.

Common Pitfalls

  • Conflating sampling bias and feature selection bias. Sampling bias is about which examples are in the training set; feature selection bias is about which features of each example are used. They have different mechanisms and different mitigations.
  • Assuming removing a single proxy variable eliminates bias. Other correlated features (commute distance, name, graduation cohort) often remain, and the model can shift its dependence onto them.
  • Treating aggregate accuracy as evidence of fairness. A 95% aggregate score can hide a 99/70 subgroup split. Disaggregated evaluation is what reveals the gap.
  • Treating computing bias as a moral failing of the developers rather than a statistical property of the data and the procedure. The textbook frames bias technically: it arises from the data, the features, and the evaluation procedure, and is addressed by technical mitigations.
  • Believing that non-machine-learning systems are immune. A search ranker using historical click-through, or a recommender optimizing engagement time, can amplify historical biases through the same statistical mechanism.

Connect to the Exam

The AP exam tests three competencies on computing bias: (a) identify whether a scenario describes data bias, algorithm-design bias, or interaction between the two; (b) describe at least one mechanism producing each type; and (c) name at least one corresponding mitigation. Expect short scenarios — a spam filter, a loan model, a facial-recognition system, a search ranker — where you must classify the mechanism and pair it with the right technical response. Use the textbook's exact mechanism-mitigation vocabulary: sampling bias -> rebalancing, proxy variable -> feature auditing, evaluation gap -> subgroup metrics + model card. Avoid policy framing; the exam tests technical reasoning about mechanisms and mitigations, not normative judgments.

Self-Check

  • A medical-imaging model is trained on data where 80% of images come from one hospital system. Identify the bias mechanism most likely to arise at deployment to a different hospital, and name the standard technical mitigation.
  • Explain in two sentences why simply removing race as an input feature is not sufficient to eliminate disparate outcomes in a model that uses ZIP code, first name, and commute distance.
  • A model achieves 92% aggregate accuracy. Disaggregated evaluation shows 96% on Group A and 78% on Group B. Identify the mechanism this disparity illustrates and the procedural mitigation.
  • The textbook says computing bias also arises in non-machine-learning systems. Construct one example (search, recommendation, ranking) and trace how biased historical data feeds into biased outputs without any model training step.
  • A model card lists training-data provenance, evaluation methodology, and per-subgroup performance. Explain why each of these three sections matters to a downstream user deciding whether to deploy the model in a new context.
