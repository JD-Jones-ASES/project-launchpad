AP CSP — Topic 9.2: The Code Requirements Checklist

Core Question

What exactly must your Create Performance Task program contain, and where must each required element live? The Program Requirements section of the Task Guidelines lists the requirements in plain language, but the rubric scores them with surprising precision: a loop in main() does not count as a loop in the procedure, a parameter that is never used is the same as no parameter, and a "list of items" stored as three separate variables is no list at all. Section 9.2 converts the rubric language into a nine-row checklist you can run against your code before submission, and it identifies the five auto-disqualifying mistakes that mechanically lose points regardless of how good the rest of the submission is.

Key Concepts

  • The Five Mandatory Elements: input, output, list, student-developed parameterized procedure, and sequencing+selection+iteration inside the procedure body. Every element must be present and properly placed.
  • Input Requirement: At least one value accepted from the user — input(), file read, button click, GUI form. Hard-coded constants do not qualify.
  • Output Requirement: At least one result displayed or returned that is meaningful to the user. Visible to the reader and demo.
  • List Requirement: A list (or other collection) created AND used. Empty declarations that are never populated do not count. Append-and-iterate is the canonical pattern.
  • Student-Developed Parameterized Procedure: A def the student wrote, accepting at least one parameter and using that parameter meaningfully. Library imports and unmodified AI-generated procedures fail the "student-developed" qualifier.
  • Sequencing in the Procedure Body: Multiple distinct ordered steps inside the procedure. compute_streak does four: filter, dedupe+sort, walk for longest run, return.
  • Selection in the Procedure Body: An if/elif/else inside the procedure that decides behavior based on a condition. The filter if subject == target_subject and minutes >= 30: is the canonical example.
  • Iteration in the Procedure Body: A for or while loop inside the procedure. compute_streak contains two: the filter loop and the streak-counting loop.
  • Different-Argument Demonstration: The procedure must be called with at least two different argument values during the video demo, producing two different observable outputs.
  • Auto-Disqualifying Patterns: Five named patterns (no parameter, constructs outside the procedure, no-op parameter, fake list, no observable output) cause automatic point loss on the algorithm and abstraction rows.

PT Rubric Alignment

Section 9.2 maps directly onto five of the six PT rubric rows. The Data Abstraction row scores presence of an actual list and explanation of how it manages complexity — Auto-disqualifier D applies here. The Managing Complexity and Procedural Abstraction rows depend on the parameterized procedure, with Auto-disqualifiers A (no parameter) and C (no-op parameter) blocking credit. The Algorithm Implementation row requires sequencing+selection+iteration inside the procedure body — Auto-disqualifier B blocks it when constructs live elsewhere. The Testing row depends on observable output — Auto-disqualifier E blocks it when the procedure does nothing visible. Only the Program Purpose and Function row (covered in Topic 9.1) is about Written Response language rather than code structure; the other five depend on whether your submitted code passes this checklist mechanically.

Common Pitfalls

  • Iteration in main(), not the procedure. A common pattern: students write the procedure as just an if statement and put the for loop in main() to iterate over records and call the procedure once per record. This fails Auto-disqualifier B. Move the loop inside the procedure body and pass the entire list as a parameter.
  • A parameter named but never used. Students sometimes add a parameter "to satisfy the rubric" but write the procedure as if it were a constant. Auto-disqualifier C catches this. Make sure your parameter actually drives a comparison, an index access, or a return-value path inside the procedure.
  • Three variables masquerading as a list. Storing item1, item2, item3 and calling them a list in the Written Response is the textbook Auto-disqualifier D failure. Use an actual Python list (or dict, set, tuple-of-tuples).
  • Procedure that returns nothing and prints nothing. A procedure that mutates a local and exits is invisible to the reader. Auto-disqualifier E. Either return a value the caller uses, or print directly so the demo shows visible output.
  • Calling the procedure with the same argument twice. Two calls of compute_streak(records, "math") do not demonstrate procedural abstraction — both produce the same output. Use at least two different argument values to prove the procedure abstracts over its parameter.

Connect to the Exam

This section underpins the entire Code Submission portion of the Create PT, plus the Algorithm Development and Data and Procedural Abstraction categories of the in-exam Written Response. Run the nine-row checklist against your code at least one week before the submission deadline so any pattern violations can be repaired without rushing. On exam day, the Written Response prompts will ask you to explain how your procedure uses sequencing, selection, and iteration — those words must match constructs the reader can locate in your PPR. If your code passes the checklist, your Written Response will have unambiguous evidence to cite. If it fails the checklist, no Written Response language can recover the lost points.

Self-Check

  • A student writes def find_max(data): that initializes best = 0 and returns it without referencing data. Which auto-disqualifier triggers? Explain in one sentence what the rubric requires the parameter to do.
  • A student stores three grocery items as item1 = "apples", item2 = "bread", item3 = "cheese" and calls this "a list of grocery items" in the Written Response. Will the data-abstraction point be awarded? Why or why not?
  • Look at your own submission. For each of the five mandatory elements, identify exactly which line(s) of code provide the evidence. If you cannot point to a specific line for any one element, the rubric row is at risk.
  • A classmate's procedure is structured so that all sequencing and selection happen inside the procedure but the iteration happens in main() (the loop calls the procedure once per record). Diagnose the auto-disqualifier and propose a refactor that moves iteration inside while preserving the program's behavior.
  • Trace one call to compute_streak(study_records, "math") through the four sequencing phases inside the procedure. Identify the specific lines that demonstrate sequencing, selection, and iteration. Could you cite each of these by line number in a Written Response under exam-day time pressure?
