AP CSP — Topic 6.3: Algorithmic Efficiency and Heuristics

Core Question

Why do two correct algorithms for the same problem differ enormously in usefulness, and what AP-named tools — polynomial vs exponential growth, reasonable vs unreasonable time, exact algorithms vs heuristics — describe that gap?

Key Concepts

  • An algorithm's efficiency describes how its number of operations grows as the input size n grows.
  • Reasonable time covers polynomial growth: O(1), O(log n), O(n), O(n^2), and other O(n^k) for fixed k. These algorithms remain practical even at large n.
  • Unreasonable time covers exponential growth (O(2^n) or worse). Each additional input element doubles the work, so the runtime quickly exceeds any human-relevant timescale.
  • O(1) is constant; O(log n) rises barely noticeably; O(n) grows in a straight line; O(n^2) bends upward and diverges quickly; O(2^n) is the cliff that dwarfs every polynomial.
  • The "exponential cliff" intuition: at n = 60, an O(2^n) algorithm needs ~10^18 operations — about 36 years on a 10^9 ops/sec computer. Adding one more element doubles that to 73 years.
  • Hardware improvements only shift the cliff slightly. Doubling CPU speed buys you exactly one extra element; the exponential curve quickly out-grows any constant factor.
  • An intractable problem is one for which no known algorithm solves every instance in polynomial time. The Traveling Salesman Problem (TSP) and many scheduling problems are intractable.
  • A heuristic sacrifices guaranteed optimality in exchange for a "good enough" answer in reasonable time. Heuristics are how engineers cope with intractable problems.
  • Real-world heuristics include nearest-neighbor TSP (logistics routing), A* search (GPS navigation, game pathfinding), and simulated annealing (circuit-board layout, protein folding).
  • The AP exam does not require formal Big-O computation. It does require classifying algorithms by growth class and recognizing the reasonable vs unreasonable distinction.

Worked Examples / Code Walkthroughs

Example 1 — Comparing growth rates at n = 60

Class Operations at n = 60 At 10^9 ops/sec
O(1) 1 instant
O(log n) ~6 instant
O(n) 60 60 nanoseconds
O(n^2) 3,600 3.6 microseconds
O(2^n) ~1.15 * 10^18 ~36.5 years

The polynomial classes finish before you can blink. The exponential class takes longer than a human career. This is the practical meaning of "reasonable" vs "unreasonable" time.

Example 2 — Why pairwise comparison is O(n^2)

def has_duplicate(items):
    for i in range(len(items)):
        for j in range(i + 1, len(items)):
            if items[i] == items[j]:
                return True
    return False

The outer loop runs n times. For each iteration of the outer loop, the inner loop runs up to n times. Total comparisons grow as roughly n*n = n^2. Doubling the input size quadruples the runtime — the textbook quadratic pattern.

Example 3 — Nearest-neighbor TSP heuristic

def nearest_neighbor_tsp(cities, distances):
    """Greedy heuristic: always go to the nearest unvisited city.
       Returns a route list - not guaranteed optimal, but fast.
    """
    unvisited = list(cities)
    route = [unvisited.pop(0)]              # start at the first city

    while len(unvisited) > 0:               # iteration
        current = route[-1]
        nearest = None
        shortest_dist = float("inf")
        for city in unvisited:              # find nearest unvisited
            if distances[current][city] < shortest_dist:   # selection
                shortest_dist = distances[current][city]
                nearest = city
        route.append(nearest)               # sequencing
        unvisited.remove(nearest)
    return route                            # good-enough solution in O(n^2)

Two important features. First, the procedure body has sequencing, selection, AND iteration — exactly the Create PT rubric structure. Second, it runs in O(n^2) (outer while * inner for), which is reasonable time. The route it produces is typically within 20-25% of optimal — a trade-off that beats waiting months for the exact answer.

Common Pitfalls

  • Treating "reasonable time" and "fast" as identical. An O(n^2) algorithm at n = 1,000,000 is slow but still polynomial-time and tractable; an O(2^30) is short to write but already unreasonable.
  • Computing Big-O formally on the exam. The AP CED only asks you to classify and compare, not derive.
  • Believing faster hardware solves exponential growth. Doubling CPU speed only buys you one more element of input.
  • Thinking heuristics are second-class. In industry (logistics, GPS, ML), heuristics are the workhorse — exact algorithms are often impossible.
  • Confusing "intractable" with "unsolvable." Intractable problems CAN be solved exactly in principle; just not quickly. Section 6.4's undecidable problems are different — those cannot be solved at all by any algorithm.

Connect to the Exam

This section maps directly to AP CED Topic 3.6 (Algorithmic Efficiency). Multiple-choice questions show two algorithms or growth classes and ask "which scales worst?" Free-response questions on the Create PT written response can ask whether your procedure runs in reasonable time and to defend that classification. The MCQ section also tests recognition of heuristics: expect a TSP- or scheduling-flavored stem with an answer mentioning "good enough in polynomial time."

Self-Check

  1. An algorithm that checks every pair of elements in a list of n items is in what Big-O class?
  2. A task requires 2^40 operations on a 10^9 ops/sec machine. About how many minutes does it take?
  3. In one sentence, why is the nearest-neighbor TSP heuristic useful even though it does not always find the shortest route?
  4. Order from fastest- to slowest-growing: O(n^2), O(1), O(2^n), O(log n), O(n).
  5. Is sorting a list with merge sort an example of a tractable or intractable problem? Justify with a Big-O class.
