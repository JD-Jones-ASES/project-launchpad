AP CSP — Topic 3.1: Boolean Expressions and Relational Operators

Core Question

How does a program decide between alternatives — granting access, displaying a discount, sending an alert — when computers ultimately operate on numbers and bits? Every decision begins by reducing a question about the world ("is the user old enough?") into a Boolean expression that evaluates to exactly True or False. This section establishes the vocabulary of Boolean values and the six relational operators that produce them, distinguishes Python's == from AP Pseudocode's =, and traces the lineage of the entire idea back to George Boole's nineteenth-century algebra of logic.

Key Concepts

  • Boolean value: A data type with exactly two values — True or False in Python, TRUE or FALSE in AP Pseudocode.
  • Boolean expression: Any expression that evaluates to a Boolean. Built from a relational operator applied to two values, or from existing Booleans combined by logical operators.
  • Relational (comparison) operator: A symbol that compares two values and returns a Boolean. Python set: ==, !=, <, >, <=, >=.
  • Pseudocode equivalents: AP Pseudocode uses =, !=, <, >, <=, >= — the critical difference being a single = for equality (not ==).
  • Predicate: A Boolean expression viewed as a testable claim. age >= 18 is the predicate "age is at least 18," returning True or False depending on the value of age.
  • Equality vs. assignment in Python: == compares; = assigns. Putting = inside an if condition (e.g. if x = 5:) raises a SyntaxError because Python forbids assignment inside an if.
  • Equality vs. assignment in AP Pseudocode: <- assigns; = compares. The arrow makes assignment unambiguous, freeing = to serve only as comparison.
  • Boundary conditions: >= includes the boundary value, > excludes it. Off-by-one errors come from confusing the two; prefer wording that mirrors the requirement (score >= 90 rather than score > 89).
  • String comparison: Python uses lexicographic (dictionary) order via ASCII. "apple" < "banana" is True because 'a' precedes 'b'. Case is significant: "apple" == "APPLE" is False.
  • TypeError on mixed types: Comparing a string to an int with < raises TypeError in Python 3. Numeric int-vs-float comparisons (3.0 == 3) are valid.

Worked Examples / Code Walkthroughs

Example 1 — Six relational operators in Python

age = 17
price = 49.95
print(age >= 18)     # False - age is NOT >= 18
print(age == 17)     # True  - age IS equal to 17
print(price < 50.00) # True  - 49.95 < 50.00
print(price != 19.99)# True  - 49.95 is not 19.99
print(age <= 17)     # True  - boundary case: equal counts

Each line evaluates a single predicate. Note how <= includes the boundary value 17, while >= excludes it (since 17 is not greater than or equal to 18).

Example 2 — The == vs = trap (Python)

x = 5         # ASSIGNMENT: stores 5 in x
# if x = 5:   # SyntaxError - assignment forbidden in if
if x == 5:    # CORRECT: tests equality
    print("equal")

A single = inside an if is a SyntaxError because Python's grammar requires a Boolean expression there, not an assignment statement.

Example 3 — Same logic in AP Pseudocode

x <- 5             // ASSIGNMENT: arrow stores 5 in x
IF (x = 5)         // EQUALITY: single = compares
{
    DISPLAY("equal")
}

In Pseudocode the arrow <- distinguishes assignment, so the single = is unambiguously a comparison. Translating between languages requires swapping <- ↔ = (assignment) and = ↔ == (equality).

Example 4 — Boundary conditions and string comparison

score = 90
print(score >= 90)         # True  (boundary inclusive)
print(score > 90)          # False (boundary excluded)

print("apple" < "banana")  # True  ('a' < 'b' in ASCII)
print("apple" == "APPLE")  # False (case matters)

>= is the standard idiom for "at least N" — it expresses intent more directly than > (N-1) and avoids off-by-one errors when the data is non-integer.

Common Pitfalls

  • Writing if x = 5: in Python: SyntaxError. Use == for comparison.
  • Forgetting that AP Pseudocode reverses the convention: Pseudocode uses single = for equality and <- for assignment. Don't translate Pseudocode IF (x = 5) into Python as if x = 5: — translate it to if x == 5:.
  • Off-by-one errors at boundaries: > 89 and >= 90 happen to agree on integers but disagree on floats like 89.5. Mirror the requirement in code: "at least 90" → >= 90.
  • Comparing across types: "5" < 5 raises TypeError in Python 3. Convert one operand explicitly with int() or str() first.
  • Case sensitivity in string comparisons: "Alex" == "alex" is False. Normalize with .lower() if case should not matter.

Connect to the Exam

Topic 3.1 supports Big Idea 3 (Algorithms and Programming) and is foundational for every later topic involving conditionals, loops, or compound logic. Expect questions that ask you to evaluate a single Python expression to True or False, translate between Python == and Pseudocode =, identify why if x = 5: is a SyntaxError, or distinguish boundary-inclusive >= from boundary-exclusive >. The College Board reference sheet uses Pseudocode notation (= for equality, <- for assignment), so during the exam you must mentally swap conventions when reading code in either language.

Self-Check

  • Given age = 18, evaluate each: age == 18, age != 18, age > 18, age >= 18. Which ones are True and why?
  • A peer writes if score = 95: in Python. What error does Python report, and what one-character change fixes the code?
  • Translate this Pseudocode into Python: IF (color = "red") { count <- count + 1 }. Where do = and <- change?
  • Without using and, write the two separate Python expressions that — when both True — mean "temp is strictly between 60 and 80."
  • Why is score >= 90 preferred over score > 89, even though they agree on integer scores? Give a concrete value where they would disagree.
