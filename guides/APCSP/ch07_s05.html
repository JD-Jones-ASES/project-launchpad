AP CSP — Topic 7.5: Sequential, Parallel, and Distributed Computing

Core Question

When does a single processor stop being enough, and what kinds of computing models can handle problems that no single processor can? Some problems — formatting a document, summing a small list — fit comfortably on one core. Others — training a modern AI model, rendering an animated film, indexing the Web — require thousands of cooperating processors. This section presents three computing models as a spectrum: sequential, parallel, and distributed. It shows what each is good for, why Amdahl's Law caps the gains from parallelism, and how cloud computing turns distributed computing into something every modern developer takes for granted.

Key Concepts

  • Sequential computing: one processor executing one instruction at a time. Simple to reason about; bounded by single-processor speed; runtime scales linearly with workload.
  • Parallel computing: multiple processors (or cores) on the same machine running independent sub-tasks simultaneously. Reduces wall-clock time for naturally decomposable workloads.
  • Independence requirement: only tasks that do not depend on each other's results can truly run in parallel. If task B must wait for task A, they are inherently sequential.
  • Distributed computing: many independent machines cooperating over a network. Each machine has its own CPU, memory, and storage; they exchange messages. Brings new challenges: network latency, partial failures, coordination overhead.
  • Amdahl's Law (conceptual): every program has an inherently sequential portion that limits the gains from parallelism. If a fraction f is sequential, the maximum possible speedup is 1/f, regardless of processor count.
  • Multi-core CPUs: 4-32 cores on a single chip; the OS scheduler maps threads to cores. A web browser routinely uses multiple threads (rendering, JS, network I/O, UI).
  • GPUs: thousands of cores; originally for video-game graphics; now the workhorses of machine learning because neural-network training is essentially matrix multiplication, which is naturally parallel.
  • Cloud computing: on-demand computing, storage, and databases provided by Google, Amazon, Microsoft. Distributed computing is the foundation; the user does not see the individual machines.
  • Volunteer distributed computing: SETI@home, Folding@home — projects that aggregate idle CPU cycles from millions of personal computers worldwide.
  • The spectrum trade-off: each model trades simplicity for scale. Sequential is simplest; distributed is most powerful but most complex.

Worked Examples / Mechanisms

  • Sequential sum, walked through. Take a 1-million-element list. A simple loop total = 0; for n in numbers: total += n runs on one core, performing 1,000,000 additions in sequence. On a fast CPU this completes in milliseconds. But scale to 1 billion elements: the same approach takes roughly 1,000x as long because runtime scales linearly with workload, and a single processor cannot exceed its own speed.
  • Naturally parallel: matrix multiplication. The product of two 1,000 × 1,000 matrices has 1,000,000 output elements; each is the dot product of one row and one column. The dot products are independent of each other, so a GPU with thousands of cores can compute many simultaneously. This is why GPUs — originally designed for game graphics — became the workhorse of neural-network training: training is essentially continuous matrix multiplication.
  • Amdahl's Law worked through (no formula required). Suppose a program is 50% sequential. Total runtime is 1.0 unit, broken into 0.5 sequential and 0.5 parallel. With infinite processors, the parallel portion shrinks to nearly 0; the sequential portion stays 0.5. Maximum speedup = 1.0 / 0.5 = 2x. Now suppose 90% is sequential: maximum speedup = 1.0 / 0.9 ≈ 1.11x. The sequential portion is the bottleneck. The lesson: before buying more processors, refactor to reduce the sequential portion.
  • Distributed example: 100 TB astronomical dataset. A researcher wants to find unusual stars in 100 TB of data. Sequential is hopeless (years of work). A single multi-core machine cannot hold or process 100 TB. Distributed computing — many machines with distributed storage (like Google File System) and distributed processing frameworks (MapReduce-style) — is the right model. The data is partitioned across machines; each machine processes its partition; results are aggregated.

Common Pitfalls

  • Believing more processors always means proportionally faster programs. Amdahl's Law says no. The sequential portion of any program is the ultimate bottleneck.
  • Conflating parallel and distributed. Parallel = multiple cores in one machine, sharing memory. Distributed = multiple machines, communicating over a network. Distributed computing brings concerns parallel does not — network latency, partial machine failures, coordination overhead.
  • Treating sequential computing as obsolete. Most everyday programs run sequentially. Sequential is the right model when the problem is small, the tasks are interdependent, or simplicity beats scale.
  • Assuming a workload is parallel just because it processes a large dataset. Independence of sub-tasks is the requirement. Computing a running total of a list is not parallel — each new total depends on the previous. Computing the dot product of every (row, column) pair in matrix multiplication is parallel.
  • Confusing GPUs with general-purpose CPUs. A GPU has thousands of simple cores optimized for parallel arithmetic; a CPU has fewer cores optimized for fast sequential execution. Each is the right tool for different workloads.

Connect to the Exam

Section 7.5 supports questions that test whether you can (a) identify which computing model fits a given problem, (b) reason about Amdahl's Law without using a formula, and (c) name a real-world example of natural parallelism. Expect prompts like "A program is 80% parallelizable; what is the maximum speedup?" — work it out by reasoning about the 20% sequential ceiling. Be ready to choose sequential, parallel, or distributed for a scenario described in plain English (size of data, dependencies between tasks, geographic constraints), and to justify the choice in one or two sentences.

Self-Check

  • A program is 80% parallelizable and 20% inherently sequential. What is the maximum speedup achievable with any number of processors? Explain without using a formula.
  • Give one real-world example of a task that is naturally parallel, and explain in two sentences why it is parallel — what makes the sub-tasks independent.
  • A researcher wants to analyze a 100-terabyte astronomical dataset for signs of unusual star behavior. Which computing model is most appropriate — sequential, parallel, or distributed — and why?
  • Describe one specific challenge that distributed computing introduces but parallel computing on a single machine does not face.
  • Why are GPUs preferred over CPUs for training modern neural networks, given that CPUs typically have higher per-core clock speeds?
