AP CSP — Topic 5.5: Search Algorithms

Core Question

Why does binary search dramatically outperform linear search on sorted data, and what is the cost of getting that performance? The two search algorithms are at opposite ends of the universality-vs-speed trade-off: linear search works on any list but examines every element in the worst case; binary search runs in logarithmic time but only on sorted lists. Understanding when each is appropriate — and when the up-front cost of sorting is worth it — is the goal of this section, and it is one of the most testable algorithmic ideas on the AP CSP exam.

Key Concepts

  • Linear search: Examines each element from first to last until the target is found or the list is exhausted. Works on unsorted or sorted data. Worst-case O(n) comparisons.
  • Binary search: Compares the target to the midpoint of a sorted list and discards the half that cannot contain it. Repeats until found or empty. Worst-case O(log_2 n) comparisons.
  • Required precondition for binary search: The list MUST be sorted. The algorithm's invariant ('midpoint > target means target is in left half, else right half') only holds on sorted data.
  • Sentinel return: Both algorithms return -1 when the target is not found. -1 is outside the valid index range.
  • Halving the search space: Each binary-search step eliminates half the remaining candidates. Number of halvings to reduce n to 1 is log_2(n): 1024 -> 10, 1,048,576 -> 20.
  • Big-O: Notation describing how operation count grows with input size. O(n) is linear; O(log_2 n) is logarithmic; O(n log n) is the cost of efficient sorting.
  • Amortization argument: Sorting costs O(n log n). Binary search is worth the sort only when the same sorted data will be searched many times — the sort cost amortizes across multiple searches.
  • When linear wins: Unsorted data with few searches; very small lists where the constant factors dominate.
  • Off-by-one in binary search bounds: Python uses low = 0, high = len(lst) - 1; AP Pseudocode uses low <- 1, high <- LENGTH(lst). Both ends shift by one between languages.
  • Tony Hoare: Inventor of quicksort (1959-1960), Hoare logic, and the CSP process calculus. Turing Award (1980). Famously called his introduction of the null reference 'my billion-dollar mistake.'

Worked Examples / Code Walkthroughs

Linear search

def linear_search(lst, target):
    """Return the index of target in lst, or -1 if not found."""
    for i in range(len(lst)):
        if lst[i] == target:
            return i        # found at position i (0-based)
    return -1               # not found

numbers = [34, 7, 23, 32, 5, 62]
print(linear_search(numbers, 23))   # 2
print(linear_search(numbers, 99))   # -1

The loop touches every element until a match is found. Worst case (target absent): n comparisons.

Binary search

def binary_search(lst, target):
    """Return the index of target in SORTED lst, or -1 if not found.
    REQUIRES: lst must be sorted in ascending order."""
    low = 0
    high = len(lst) - 1

    while low <= high:
        mid = (low + high) // 2     # integer division for midpoint
        if lst[mid] == target:
            return mid              # found
        elif lst[mid] < target:
            low = mid + 1           # target in right half
        else:
            high = mid - 1          # target in left half

    return -1                       # not found

data = [1, 2, 3, 5, 7, 10, 12, 14, 17, 19, 22, 25, 28, 31, 36, 40]
print(binary_search(data, 22))   # 10
print(binary_search(data, 99))   # -1

For n = 16 the worst case is log_2(16) = 4 comparisons.

Comparing comparison counts (n = 1,000,000)

Linear search worst case:  1,000,000 comparisons   (O(n))
Binary search worst case:         20 comparisons   (O(log_2 n), since log_2(10^6) is approximately 20)

The asymmetry is dramatic — but only the sorted-data prerequisite is met.

Pseudocode bounds for binary search

// AP Pseudocode - 1-based indexing
low <- 1
high <- LENGTH(lst)
REPEAT UNTIL (low > high)
    mid <- (low + high) / 2     // integer division
    IF (lst[mid] = target)
        RETURN mid
    IF (lst[mid] < target)
        low <- mid + 1
    ELSE
        high <- mid - 1
RETURN -1

Both bounds shift by one from Python: low starts at 1 (not 0), high starts at LENGTH(lst) (not LENGTH(lst) - 1).

Common Pitfalls

  • Running binary search on unsorted data: The function will run to completion but return unreliable results. Python does NOT auto-sort. The programmer must guarantee sorted input.
  • Forgetting the sort cost: A single binary search on unsorted data is slower than a single linear search because sorting is O(n log n). The amortization argument requires multiple searches against the same sorted list.
  • Off-by-one in initial bounds: Python's high = len(lst) - 1 vs. Pseudocode's high <- LENGTH(lst). Mixing them causes index errors or missed elements.
  • Confusing average and worst case for linear search: Average is about n/2; worst case is n. Big-O bounds describe the worst case.
  • Forgetting the sentinel return: A function that does not return -1 when the target is absent produces undefined behavior. Always handle the not-found case explicitly.

Connect to the Exam

AP CSP exam items on search algorithms test the trade-off explicitly: a stimulus describes a sorted vs. unsorted list and asks which algorithm is appropriate, or asks how many comparisons binary search needs for n = 256 (answer: log_2(256) = 8). The Create PT 'Algorithm' rubric row often features a search or filter pattern, and you will need to articulate why the algorithm is correct. Binary search vs. linear search is also a frequent multiple-choice topic — the trap is to forget the sortedness prerequisite or to claim binary search is always better.

Self-Check

  1. A sorted list has 256 elements. What is the maximum number of comparisons binary search will make? Show your work using log_2(n).
  2. Name ONE situation where linear search is the better choice over binary search, even for a large list.
  3. In AP Pseudocode, what is the first index binary search should set as low? What is the first index it should set as high?
  4. Explain in one sentence why running binary search on an unsorted list returns unreliable results.
  5. For n = 1,000,000, give the worst-case number of comparisons for linear search and for binary search. Why is the gap so large?
