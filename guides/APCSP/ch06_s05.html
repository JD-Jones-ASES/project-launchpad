AP CSP — Topic 6.5: Capstone: Building the Study Streak Tracker

Core Question

How does a single program — the Study Streak Tracker — satisfy all four Create Performance Task code requirements, and how do you write the function-and-purpose response that earns the matching written-response rubric points?

Key Concepts

  • The Create PT has four code requirements: (1) input, (2) a list used to manage complexity, (3) a student-developed parameterized procedure whose body contains sequencing, selection, and iteration, and (4) output to the user.
  • Tracker v2 satisfies all four in one short program: append-style data entry, the study_records list, compute_streak as the procedure, and print statements as output.
  • The procedure compute_streak(records, target_subject) takes two parameters. Its body filters qualifying dates, sorts and deduplicates, then walks the list to compute the longest consecutive-day streak.
  • The PT also grades a Function & Purpose row in the written response: function describes WHAT the procedure computes; purpose describes WHY a user would want it. Both are required for full credit.
  • A common Chief Reader complaint is students who write only the function and skip the purpose, losing the row.
  • compute_streak is pure: same inputs, same output, no side effects. That makes the procedure trivially testable with sample data.
  • The procedure body uses date.fromisoformat() to convert ISO date strings to date objects so consecutive-day differences are computed cleanly via subtraction.
  • An empty-input early return (return 0 when qualifying_dates is empty) prevents downstream errors when the target subject has no qualifying records.
  • The same procedure works for any subject just by changing the argument at the call site — this is the abstraction win that justifies extracting the procedure.
  • AP Pseudocode equivalents: def becomes PROCEDURE, the colon and indentation become curly braces, return becomes RETURN, and = at call sites becomes <-.

Worked Examples / Code Walkthroughs

Example 1 — The full Tracker v2 (rubric-checklist version)

from datetime import date

# [1] INPUT — data enters via append() calls (full PT version uses input())
study_records = []
study_records.append(("2025-09-01", "math", 45))
study_records.append(("2025-09-01", "history", 30))
study_records.append(("2025-09-02", "math", 60))
study_records.append(("2025-09-02", "history", 25))
study_records.append(("2025-09-03", "math", 35))
study_records.append(("2025-09-04", "math", 50))
study_records.append(("2025-09-05", "history", 40))
study_records.append(("2025-09-06", "math", 30))

# [2] PARAMETERIZED PROCEDURE with sequencing + selection + iteration:
def compute_streak(records, target_subject):
    qualifying_dates = []
    for record in records:                              # iteration
        rd, subject, minutes = record
        if subject == target_subject and minutes >= 30:  # selection
            qualifying_dates.append(rd)                  # sequencing
    if len(qualifying_dates) == 0:
        return 0
    unique_dates = sorted(set(qualifying_dates))
    longest = 1
    current = 1
    for i in range(1, len(unique_dates)):                # iteration
        prev = date.fromisoformat(unique_dates[i-1])
        cur = date.fromisoformat(unique_dates[i])
        if (cur - prev).days == 1:                       # selection
            current += 1
            if current > longest:
                longest = current
        else:
            current = 1
    return longest

# [3] LIST used to manage complexity (without the list, we'd need
# a separate variable for every day of study)

# [4] OUTPUT — result displayed to the user
print("Math streak: " + str(compute_streak(study_records, "math")) + " days")
print("History streak: " + str(compute_streak(study_records, "history")) + " days")
print("Spanish streak: " + str(compute_streak(study_records, "spanish")) + " days")

Expected output:

Math streak: 4 days
History streak: 1 days
Spanish streak: 0 days

Example 2 — Tracing the math streak

Math entries with at least 30 minutes: 09-01 (45), 09-02 (60), 09-03 (35), 09-04 (50), 09-06 (30). After sorting and deduping: [09-01, 09-02, 09-03, 09-04, 09-06]. Walking the list:

  • i=1: cur 09-02 - prev 09-01 = 1 day -> current=2, longest=2
  • i=2: cur 09-03 - prev 09-02 = 1 day -> current=3, longest=3
  • i=3: cur 09-04 - prev 09-03 = 1 day -> current=4, longest=4
  • i=4: cur 09-06 - prev 09-04 = 2 days -> reset current=1, longest stays 4

Returned value: 4. The history result (1) and spanish result (0) follow from the same logic on their respective filtered lists.

Example 3 — The Function-vs-Purpose written response

Function (one sentence): "compute_streak iterates over a list of study records, filters for the target subject with at least 30 minutes, and returns the length of the longest consecutive-day streak."

Purpose (one sentence): "compute_streak helps students identify which subjects they are studying consistently, so they can detect gaps before an exam and adjust their schedule."

Both sentences are required for the rubric row. Function describes computation; purpose describes user benefit. Practice writing both on every procedure you create.

Example 4 — AP Pseudocode equivalents

Header:

PROCEDURE compute_streak(records, target_subject)
{
    ... body in curly braces ...
}

Call site:

result <- compute_streak(study_records, "math")

Replace def with PROCEDURE, the colon with curly braces, return with RETURN inside the body, and = with <- at call sites. The argument order is identical — argument-to-parameter mapping is positional in both notations.

Common Pitfalls

  • Writing only the function description (what the code does) and forgetting the purpose (why a user wants it). The Function & Purpose rubric row requires BOTH.
  • Putting the for loop and if statements outside the procedure and only returning the count inside. The rubric requires all three constructs INSIDE the procedure body.
  • Forgetting the empty-input check. Without if len(qualifying_dates) == 0: return 0, code that calls date.fromisoformat on an empty list raises an error.
  • Reversing argument order at the call site (e.g., compute_streak('math', study_records)). Argument-to-parameter mapping is positional; the procedure receives wrong data and Python cannot detect it.
  • Hardcoding the subject inside the procedure body. The whole point of v2 is to parameterize the subject so the procedure works for any.

Connect to the Exam

The Create PT (worth 6 of the 70 total exam points) is graded against this exact requirement list. Tracker v2 is the single example that ties every Unit 6 concept together: defining a parameterized procedure (6.1), parameter/argument/return-value mechanics (6.2), and a body whose efficiency is reasonable-time (6.3, two linear passes plus a sort). Free-response questions on the Function & Purpose row directly grade the language modeled in Example 3 of this guide. Practice writing both halves of the response for every procedure you build — the Chief Reader Report keeps flagging single-half answers as the most common reason students lose this row.
