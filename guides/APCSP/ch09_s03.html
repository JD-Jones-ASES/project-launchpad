AP CSP — Topic 9.3: The Video Demonstration

Core Question

How can sixty seconds of silent screen recording demonstrate everything the Create PT rubric requires of a working program — and what mistakes turn an otherwise excellent demo into an automatic zero? The Create PT video is the artifact most often torpedoed by an avoidable rule violation. The College Board's requirements are short, absolute, and binary: a 61-second video is rejected outright; a video that contains a single voice-narration phrase is rejected; a video showing the student's name in a window title bar is rejected. Section 9.3 walks through every constraint, identifies the four most common pitfalls, and provides a 60-second budget plan that lets the Study Streak Tracker satisfy every video-related rubric requirement.

Key Concepts

  • Length Limit (60 seconds): A 61-second video earns automatic zero on the video rubric rows. Enforced by upload validation and human readers. No partial credit.
  • File Size Limit (30 MB): Independent of length. Re-encode with HandBrake if necessary.
  • Format Requirement: MP4, MOV, or WMV. Check current Task Guidelines for the latest list.
  • Content: Input AND Output: The video must show the user entering at least one value AND the program producing the resulting output. Both visible in the same recording.
  • No Narration: Voice-over commentary is prohibited. Ambient sound (keyboard, room) is acceptable; silence is preferable. Disable the microphone.
  • No Identifying Information: Student name, school name, teacher name, username, email address — none may appear in the video frame.
  • Different-Argument Demonstration: The procedure must be called at least twice with different arguments, producing different observable outputs. This proves procedural abstraction in the video itself.
  • Title-Card Overshoot Pitfall: Adding intro/outro cards to a 58-second demo can push final duration past 60 seconds. Plan content under length, not at length.
  • Output Legibility: Pre-recording configuration of terminal font (16-18 pt), color scheme (high contrast), and recording-area framing prevents the most common fatal content error.
  • The 60-Second Budget: Time-allocation table mapping each segment to a rubric requirement, ensuring every required demonstration fits within the limit.

PT Rubric Alignment

The video supports several rubric rows simultaneously. Procedural Abstraction is the row most directly tied to the video — calling the procedure with two different arguments and showing two different outputs is the single most efficient way to provide direct evidence for that row. Algorithm Implementation also benefits when the visible output reflects what the algorithm computes (different streak counts for "math" and "spanish" prove the iteration and selection inside the procedure are working correctly with different filter conditions). Testing is implicitly supported by visibly running the program with multiple inputs. The video also feeds the Program Purpose and Function row by showing that the program actually does what the Written Response claims it does. Note that the video itself does not directly score the Purpose row — the language in the Written Response does — but a visible disconnect between the demo and the Written Response can erode reader trust on every dependent row.

Common Pitfalls

  • Title-card overshoot. Students record a 58-second demo, then add a 5-second intro and a 5-second outro, and submit a 68-second file. Plan total file length, not just demo content. Record so total final length is 50-58 seconds with comfortable buffer.
  • Output legibility failure. Default terminal fonts and color schemes are not designed for downstream video playback. Enlarge to 16-18 pt and use high contrast (white on black, or black on white). Frame the recording area to capture the full terminal from prompt to last output line. Verify legibility on a different device after rendering.
  • Accidental voice narration. Many screen-recording tools enable the microphone by default. A single "okay, now I'll type a subject" disqualifies the video. Disable the microphone before pressing record and verify no audio is captured by playing back briefly.
  • IDE file path leakage. The most-missed identifying-information violation. The IDE chrome shows C:\Users\jane.smith\... at the top of the editor pane. Either hide the file-tree panel, work in a directory that doesn't reveal username, or crop the recording area to exclude the path bar.
  • Single procedure call. Calling the procedure once does not demonstrate procedural abstraction in the video. Two calls with different arguments and different outputs is the rubric-aligned pattern; sometimes called the "math then spanish" demonstration in this textbook.

Connect to the Exam

The video is one of three submitted artifacts (code, video, PPR). Sections 9.1, 9.2, and 9.4 cover the others; Section 9.3 ensures the video does its job. On exam day the in-exam Written Response cannot reference the video directly (the video is not visible to the student during the exam — only the PPR is), but the video's evidence affects the rubric scoring of every program-behavior claim made in the Written Response. The most reliable preparation is to record a draft video at least one week before submission, watch it on a different device than you recorded on (verifying legibility under typical reader conditions), and run the binary checklist: ≤60 sec? ≤30 MB? Input shown? Output shown? No narration? No identifying info? Different-argument calls visible? Six checkmarks before final submission.

Self-Check

  • A student finishes recording and the file is 62 seconds long. Name two strategies they could use to reduce the duration without re-recording from scratch. (Hint: think trim, not cut required content.)
  • Explain why calling compute_streak() twice with different arguments during the video — instead of once — is strategically important for the rubric, not just for demonstration clarity.
  • Audit the IDE you currently use. Does it display your username, school email, or any identifying file path in any panel that would appear in a screen recording? Sketch a recording-area crop that excludes those zones while still capturing the terminal output.
  • Suppose your demo runs cleanly but you realize the terminal default theme (gray on dark gray) is hard to read at 720p playback. Describe the steps to reconfigure the terminal for high-contrast capture without altering the source code or program behavior.
  • Plan a 60-second time-allocation table for your own Create PT program. List each segment, the rubric requirement it satisfies, and the seconds budgeted. Build in a 5-second buffer to absorb any over-runs. Compare against the Study Streak Tracker plan in Section 9.3.
