AP CSP — Topic 4.5: Random Numbers and Simulation

Core Question

How can a deterministic computer produce behavior that appears random — and why is that useful? Many of the most important computational problems (weather modeling, game design, statistical sampling, scientific simulation, cryptography) require unpredictable behavior. The solution is pseudo-random number generation: an algorithm that produces a sequence of numbers passing statistical randomness tests, even though the sequence is fully determined by an initial seed. Python's random module supplies the tools, AP Pseudocode's RANDOM(a, b) is the exam-side analog, and combining randomness with loops yields simulation: estimating properties of a system by running it many times. Mastering Topic 4.5 means knowing the inclusive-on-both-ends rule for randint/RANDOM, the law of large numbers, and when randomness is and is not appropriate.

Key Concepts

  • Pseudo-random number generator (PRNG): An algorithm that produces a deterministic but statistically-random-looking sequence, seeded by an initial value (typically system time).
  • random.randint(a, b): Python function returning an integer from a to b INCLUSIVE on both ends. random.randint(1, 6) → 1, 2, 3, 4, 5, or 6 with equal probability.
  • random.random(): Python function returning a float in [0.0, 1.0) — at least 0.0, strictly less than 1.0. Used for probability decisions like if random.random() < 0.3.
  • AP Pseudocode RANDOM(a, b): Returns an integer uniformly at random from a through b inclusive on both ends. Direct match to Python's random.randint(a, b).
  • Inclusive on both ends rule: Unlike range (which excludes the upper bound), randint and RANDOM both include both endpoints. This is the rare case where Python and AP Pseudocode agree, so no shift is required when translating.
  • Simulation: Running an experiment many times in software, tallying results. Combines a loop with random number generation.
  • Law of large numbers: As trials grow, empirical proportions converge to theoretical probabilities. Small N has large variance; large N has tight convergence.
  • Monte Carlo method: Estimating a quantity by random sampling — throw darts at a unit square, count those inside an inscribed quarter-circle, multiply by 4 to estimate π.
  • When randomness is appropriate: Simulation, games, statistical sampling, modeling. Reproducibility achievable by seeding the PRNG.
  • When randomness is NOT appropriate: Security (passwords, tokens, keys) — use Python's secrets module instead, which uses OS entropy. Also: when reproducibility is required, seed the generator with a fixed value.

Worked Examples / Code Walkthroughs

Example 1 — Inclusive-on-both-ends: dice and coins

import random

# Six-sided die roll: 1, 2, 3, 4, 5, or 6
die = random.randint(1, 6)

# Coin flip: 0 (tails) or 1 (heads)
flip = random.randint(0, 1)

# AP Pseudocode equivalents:
die <- RANDOM(1, 6)
flip <- RANDOM(0, 1)

Both forms include both endpoints. RANDOM(1, 6) can return 1 or 6; range(1, 6) cannot return 6 — these are different functions, easily confused.

Example 2 — Law of large numbers via coin flip simulation

import random

for N in [10, 100, 1000, 10000]:
    heads = 0
    for _ in range(N):
        if random.randint(0, 1) == 1:
            heads += 1
    print(f"N = {N:5d} | heads/N = {heads/N:.4f}")

Typical output:

N =    10 | heads/N = 0.6000
N =   100 | heads/N = 0.5200
N =  1000 | heads/N = 0.4990
N = 10000 | heads/N = 0.5013

Small samples deviate substantially (0.60 here); large samples converge tightly to 0.5. The convergence is the law of large numbers in action.

Example 3 — Two-dice sum simulation

import random

N = 10000
freq = [0] * 13       # freq[2] through freq[12]

for _ in range(N):
    roll = random.randint(1, 6) + random.randint(1, 6)
    freq[roll] += 1

# Theoretical: sum 7 has 6/36 ≈ 0.167 probability
# Simulation: freq[7] / N should be very close to 0.167
print(f"P(sum=7) ≈ {freq[7]/N:.4f}")

After 10,000 rolls, freq[7] is typically near 1,667 (≈ 1/6 of trials) — confirming the theoretical probability that 6 of 36 (die1, die2) pairs sum to 7.

Example 4 — Monte Carlo estimate of π

import random

N = 100000
inside = 0

for _ in range(N):
    x = random.random()    # [0.0, 1.0)
    y = random.random()
    if x**2 + y**2 < 1.0:  # inside the unit-circle quadrant
        inside += 1

pi_estimate = 4 * inside / N
print(f"π ≈ {pi_estimate:.5f}")
# Typically: 3.14xxx; converges slowly. 1,000,000 trials for 3 sig fig.

The unit square has area 1. The quarter-circle inside it has area π/4. So the fraction of random points landing inside, times 4, estimates π. Monte Carlo turns an analytical question into a sampling question.

Example 5 — Combining sentinel-controlled loops with randomness

import random

while True:
    n = int(input("Enter a number (-1 to quit): "))
    if n == -1:
        break
    roll = random.randint(1, 6)
    print(f"You rolled {roll}")

This combines indefinite loops (Topic 4.2), the sentinel pattern, and randomness. The if/break runs before random.randint, so -1 cleanly exits without rolling.

Common Pitfalls

  • Confusing range with randint/RANDOM: range(1, 6) excludes 6; random.randint(1, 6) and RANDOM(1, 6) include 6. For a fair die, you must use randint or RANDOM with the natural endpoints — do not write randint(1, 7) thinking randint excludes the upper bound.
  • Expecting exact convergence: With N = 10,000 coin flips, the proportion of heads is close to 0.5 but rarely equals exactly 0.5. The law of large numbers describes convergence, not equality.
  • Gambler's fallacy: After 5 heads in a row, the next flip is still 50/50. Independent random trials have no memory.
  • Using random for security: Python's random module is predictable. For cryptographic purposes (passwords, session tokens), use secrets.randbelow() or secrets.choice(), which use OS entropy.
  • Incorrect Monte Carlo geometry: For estimating π via the quarter-circle method, the test is x² + y² < 1, not x + y < 1. The geometric condition matters; getting it wrong silently produces a wrong estimate.

Connect to the Exam

Topic 4.5 is tested through code-tracing questions ("What range of values can random.randint(1, 6) return?"), translation questions (Python random.randint ↔ AP Pseudocode RANDOM), and conceptual questions about probability and simulation. The single most important fact is the inclusive-on-both-ends rule, since it differs from range. Expect at least one Monte Carlo or law-of-large-numbers question per AP exam — recognize that simulation with N trials approximates theoretical probability with error decreasing as 1/√N. Free-response questions sometimes ask you to write a sentinel-controlled simulation; combining Topic 4.2's pattern with Topic 4.5's randomness is a common exam scenario.

Self-Check

  • What is the difference between random.randint(1, 6) and random.random() * 6? Which one should you use to simulate a six-sided die, and why?
  • Write one line of AP Pseudocode that assigns a random integer from 1 to 100 to a variable named target.
  • A simulation of 10,000 coin flips reports the proportion of heads as 0.4978. Explain whether this is consistent with the law of large numbers for a fair coin.
  • For two six-sided dice, what is the theoretical probability that the sum is 12? How many of the 36 (die1, die2) pairs produce 12, and how many produce 7?
  • Why is Python's random module unsafe for generating session tokens for a website? What module should you use instead, and why?
