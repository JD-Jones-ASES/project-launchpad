AP CSP — Topic 7.3: Communication Protocols

Core Question

How does a smartphone in one country talk to a server in another, given that the two devices may be from different manufacturers, run different operating systems, sit on different physical networks, and have never previously interacted? The answer is the Internet's protocol suite — a layered family of open, standardized rules. This section tours the five protocols you must know for AP CSP: IP (addressing and routing), TCP (reliable, ordered delivery), UDP (fast, connectionless delivery), DNS (name-to-address translation), and HTTP (web request-response). The big idea is that they are layered: each protocol uses the services of the layer below and provides services to the layer above, which is why the Internet is simultaneously universal and extensible.

Key Concepts

  • Protocol: an agreed-upon set of rules letting two parties communicate. Open standards anyone can implement.
  • IP: the Internet Protocol; provides addressing and best-effort packet delivery. IPv4 is 32-bit (~4.3 billion addresses); IPv6 is 128-bit (vastly larger).
  • TCP: reliable, ordered, connection-oriented byte-stream protocol. Three-way handshake, sequence numbers, ACKs, retransmission, flow control.
  • UDP: connectionless, unreliable, best-effort datagram protocol. No handshake, no ACKs, no retransmission. 8-byte header.
  • DNS: hierarchical distributed database that maps names to IP addresses. Resolver descends through root, TLD, and authoritative servers; results are cached.
  • HTTP: application-layer protocol for web. Request specifies method (GET/POST/PUT/DELETE), path, headers; response carries status code, headers, body. Runs over TCP, port 80 (or 443 for HTTPS). Stateless.
  • HTTPS / TLS: HTTP wrapped in TLS for authentication and encryption. The padlock icon and https:// in the address bar indicate the channel is encrypted.
  • Layered architecture: applications on top of transport on top of IP on top of physical networks. Each layer is replaceable independently of the others.
  • Stateless vs stateful: HTTP is stateless (no memory between requests). Application state (login, cart) is layered on top via cookies and tokens.
  • Best-effort delivery: the IP guarantee — packets may be dropped, delayed, duplicated, or arrive out of order; reliability is provided (or not) at higher layers.

Worked Examples / Mechanisms

  • The TCP three-way handshake. Client → Server: SYN with initial sequence number x. Server → Client: SYN-ACK that acknowledges x+1 and announces server's initial sequence number y. Client → Server: ACK that acknowledges y+1. After the third message, both sides have agreed on starting sequence numbers in both directions and application data can flow. The handshake guarantees both sides are reachable, willing, and synchronized before reliable delivery begins.
  • TCP vs UDP — the protocol choice scenario. File download (1 GB ZIP): use TCP. Every byte must arrive correctly, and a few-millisecond pause for retransmission is invisible to the user. Real-time voice or video (Zoom, FaceTime): use UDP. A retransmitted voice packet 200 ms late is worse than missing one — the conversation has moved on. DNS lookup (single small query): use UDP. The query and response each fit in one packet; if the response is lost, retry from the application level. Live game state (multiplayer FPS): use UDP. Player input must reach the server within tens of milliseconds, even at the cost of dropped state updates. The unifying principle: TCP when every byte matters and timing is flexible; UDP when latency matters more than completeness.
  • A web page load step-by-step. (1) Browser parses URL https://www.example.com. (2) DNS resolves www.example.com to an IP address (descending root → .com TLD → example.com authoritative). (3) Browser opens a TCP connection to that IP on port 443 via the three-way handshake. (4) TLS handshake establishes an encrypted channel using public-key cryptography (Section 7.6). (5) Browser sends an HTTP GET / request through the encrypted channel. (6) Server responds with 200 OK and the HTML body. (7) Browser parses the HTML and repeats steps 4-6 for each image, script, and stylesheet referenced.
  • Caesar cipher analogy for symmetric vs public-key cryptography ties forward to 7.6, but the protocol layering shown here is what makes HTTPS deployable: TLS only had to be added; HTTP, TCP, and IP did not change.

Common Pitfalls

  • Saying UDP is "broken" or "worse than TCP". UDP omits guarantees on purpose. For latency-sensitive workloads, that omission is a feature. Choosing UDP for VoIP is correct engineering, not a compromise.
  • Confusing IP addresses with domain names. The IP address is the numeric routing target (192.0.2.1, 2001:db8::1); the domain name is the human-readable label (www.example.com). DNS bridges them.
  • Counting the wrong number of handshake messages. TCP open uses three messages (SYN, SYN-ACK, ACK). Two messages would not let both sides confirm sequence numbers; four messages applies to teardown (FIN/ACK), not opening.
  • Putting layers in the wrong order. DNS first, TCP next, TLS, then HTTP. HTTP can never run before its TCP connection exists, and TCP cannot target an IP before DNS supplies one.
  • Treating HTTP as stateful. HTTP itself is stateless; cookies and tokens add state at the application layer. A site that "remembers" you is doing so above HTTP, not because of it.

Connect to the Exam

Section 7.3 is a heavy stimulus-question section: expect prompts like "You are designing an X application; should you use TCP or UDP?" and "Explain the role of DNS in a web request" and "How many messages are exchanged in the TCP handshake?". The grading rubric is unforgiving on TCP-vs-UDP confusions: you must explain why the chosen protocol's guarantees match (or do not match) the workload's requirements, not just name the protocol. Memorize the three-way handshake message names (SYN, SYN-ACK, ACK), the IP-version sizes (IPv4 = 32 bits, IPv6 = 128 bits), and the layer order (DNS → TCP → TLS → HTTP).

Self-Check

  • You are building a real-time voice call application. Should you use TCP or UDP? Justify your answer in two sentences using the words "latency" and "retransmission."
  • Explain the role of DNS in loading a web page. What would happen — at the user's experience level — if DNS stopped working entirely?
  • How many messages are exchanged in a TCP three-way handshake before any application data can be sent? Name each message and what it announces.
  • A small mobile game sends 20 packets per second of player position to a server. Two of those packets are lost in transit. What does TCP do? What does UDP do? Which behavior fits the game's needs better?
  • Describe the protocol stack invoked when a browser loads https://www.example.com. Place DNS, TCP, TLS, and HTTP in the order they execute, and explain in one sentence what each contributes.
