AP CSP — Topic 10.2: Tracing Complex Logic

Core Question

Why do students who understand loops conceptually still lose points on the hardest AP CSP tracing questions? The honest answer: they trace in their head instead of on paper. The exam's most difficult algorithmic items share a common structure — several interacting variables, loops that modify multiple things simultaneously, and output that only becomes clear after working through every iteration in sequence. Mental tracing degrades after two iterations; working memory cannot reliably hold five variables across eight iterations. This section gives you the practice and the discipline to trace systematically, every time, on paper.

Key Concepts

  • Trace table: A structured worksheet — one row per loop iteration, one column per variable in scope. The act of writing each row is what catches errors that mental tracing misses.
  • Triple-nested loop iteration count: For three nested loops with iteration counts m, n, p, the innermost body executes m × n × p times. Each level multiplies; the inner loop runs to completion for every iteration of the next level out.
  • Inner-before-outer rule: When tracing, exhaust the inner counter completely before incrementing the outer counter. The output sequence reflects this order: the rightmost (innermost) value cycles fastest, the leftmost (outermost) cycles slowest.
  • Short-circuit evaluation: In cond1 AND cond2, if cond1 is false, cond2 is not evaluated. In cond1 OR cond2, if cond1 is true, cond2 is not evaluated. AP does not test this directly, but knowing it speeds tracing.
  • Multi-condition Boolean network: Expressions with five or more sub-conditions joined by AND/OR/NOT. Always evaluate each parenthesized sub-expression to a single truth value before combining — never attempt the whole chain in one pass.
  • NOT precedence: NOT applies only to its immediate operand. NOT (e > a) negates only (e > a), not anything else in the expression.
  • Robot grid state: Track three things every iteration — current position, current facing direction, and the result of CAN_MOVE(forward). MOVE_FORWARD changes position; ROTATE changes facing without moving.
  • Three bug classes: Syntax (wrong symbol/keyword, fails to parse), runtime (out-of-bounds, infinite loop, crashes during execution), logic (runs cleanly but produces wrong output). On error-trace exam questions, check in this order — syntax first, runtime second, logic last.
  • Off-by-one in binary search: low <- mid instead of low <- mid + 1 is a classic logic bug — the loop never narrows when target > data[mid], leading to infinite loop or wrong return.
  • Equality vs. assignment as a logic bug: In AP Pseudocode, high = mid - 1 (using = where <- is meant) is silently a no-op comparison. The line parses but does not update the variable.

Worked Examples / Code Walkthroughs

Example 1 — Triple-Nested Loop Trace

a <- 1
REPEAT 2 TIMES
  b <- 1
  REPEAT 2 TIMES
    c <- 1
    REPEAT 2 TIMES
      DISPLAY(a * 100 + b * 10 + c)
      c <- c + 1
    b <- b + 1
  a <- a + 1

Trace table (8 rows):

iter a b c output
1 1 1 1 111
2 1 1 2 112
3 1 2 1 121
4 1 2 2 122
5 2 1 1 211
6 2 1 2 212
7 2 2 1 221
8 2 2 2 222

The innermost variable c cycles fastest; a (outermost) cycles slowest.

Example 2 — Five-Input Boolean Network

For a=3, b=7, c=2, d=5, e=1, evaluate IF ((a > b) OR ((c < d) AND (NOT (e > a)))):

sub-expression value
a > b: 3 > 7 false
c < d: 2 < 5 true
e > a: 1 > 3 false
NOT (e > a): NOT(false) true
inner AND: true AND true true
outer OR: false OR true TRUE

Output: TRUE.

Example 3 — Buggy Binary Search

PROCEDURE binarySearch(data, target)
  low <- 1
  high <- LENGTH(data)
  REPEAT UNTIL (low > high)
    mid <- (low + high) / 2
    IF (data[mid] = target)
      RETURN(mid)
    IF (data[mid] > target)
      high = mid - 1                  <-- BUG (logic class via syntax slip)
    ELSE
      low <- mid                      <-- BUG (off-by-one logic)
  RETURN(-1)

Three bugs: (1) mid <- (low + high) / 2 may need integer division semantics in Python translation (runtime concern); (2) high = mid - 1 is an equality comparison, not assignment — should be high <- mid - 1 (logic via assignment slip); (3) low <- mid instead of low <- mid + 1 causes infinite loop when target > data[mid] and target is not present (logic).

Example 4 — Robot Grid Trace

Robot at (1,1) facing North on a 3×3 grid (boundary only):

REPEAT 3 TIMES
  IF (CAN_MOVE(forward))
    MOVE_FORWARD()
  ELSE
    ROTATE_RIGHT()
    MOVE_FORWARD()
iter position facing CAN_MOVE? action
1 (1,1) N yes move → (1,2)
2 (1,2) N yes move → (1,3)
3 (1,3) N NO (top edge) rotate right (now E), move → (2,3)

Final state: (2,3) facing East.

Common Pitfalls

  • Tracing in your head past two iterations: Working memory is unreliable beyond about two simultaneous variable states. Always write down a trace table for any loop with more than two iterations or more than two variables.
  • Jumping the outer counter prematurely: When the inner loop's last iteration completes, students often forget that the inner counter must FULLY exhaust before the outer counter steps. Re-read your trace and ensure each inner loop block completes all of its rows.
  • Confusing rotation with movement: A robot facing East after a ROTATE_RIGHT moves East on the next MOVE_FORWARD, not in the original direction. Re-read the facing column every iteration before deciding which cell the robot occupies.
  • Treating = as assignment in AP Pseudocode: high = mid - 1 parses as a comparison (and is silently discarded in expression context). The fix is high <- mid - 1. This is a logic bug rooted in a syntax slip.
  • Skipping the bug-class checklist: On error-trace questions, always look for syntax first (cheapest), runtime second, logic last. Skipping straight to logic without checking the cheap categories wastes time.

Connect to the Exam

Tracing-the-output questions appear throughout Section I, and the hardest items use exactly the patterns in this section: nested loops with output that depends on multiple variables, Boolean networks with four to six sub-expressions, and robot grid problems with conditional rules. Error-trace questions are also common — the procedure looks plausible at a glance, but contains one of the three bug classes. The grading rule on these is binary: get the output exactly right or earn zero. The best students invest 30 seconds in a trace table and move on with confidence; the rest guess and lose the question.

Self-Check

  • For a triple-nested loop with outer = 3, middle = 4, inner = 5, how many times does the innermost body execute? Show your work.
  • Evaluate (NOT (5 > 3)) AND ((10 = 10) OR (4 < 2)) step by step. What is the final truth value?
  • A robot at (2,2) facing South on a 3×3 grid executes MOVE_FORWARD(); ROTATE_LEFT(); MOVE_FORWARD(). What is its final position and facing direction?
  • Identify the three bug classes in this code and classify each: i <- 0; REPEAT UNTIL (i = 10); DISPLAY(arr[i+1]); i <- i. Which is syntax, which is runtime, which is logic?
  • Trace the output of for i in range(2): for j in range(3): print(i, j) (using Python's 0-indexed semantics). How many lines are printed and in what order?
