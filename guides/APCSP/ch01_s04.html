AP CSP — Topic 1.4: Hardware Limitations: Overflow and Roundoff

Core Question

Every CPU is a finite machine. The registers where arithmetic actually happens have a fixed bit width — typically 8, 16, 32, or 64 bits — and that fixed width creates two unavoidable limitations: an upper (and, for signed integers, lower) bound on integer magnitude, and an inability to exactly represent most real numbers. When computations cross these limits, two characteristic errors result: overflow (range exceeded) and roundoff (precision lost). They look similar in their consequences but arise from completely different causes and demand different fixes. Understanding both is essential for the AP exam, for financial software, for cryptography, and for any code that handles real-world values.

Key Concepts

  • Overflow: An arithmetic result that exceeds the range of the destination type's bits. For unsigned integers, the result wraps around (255 + 1 = 0 in 8-bit unsigned). For signed integers, sign can flip (a large positive sum becomes negative).
  • Unsigned integer range: An n-bit unsigned integer represents 0 to 2^n - 1. 8-bit max = 255; the maximum is exactly 2^n - 1, not 2^n.
  • Signed integer range (two's complement): An n-bit signed integer represents -2^(n-1) to +2^(n-1) - 1. 8-bit: -128 to +127 (asymmetric).
  • Wraparound: The default hardware behavior on overflow: the result is the true value modulo 2^n. The carry bit is discarded.
  • IEEE 754: The 1985 standard for floating-point arithmetic. Defines single (32-bit), double (64-bit), and quadruple (128-bit) precision formats. Standardizes rounding modes, special values (infinity, NaN), and exception handling.
  • IEEE 754 format: A floating-point value is sign * significand * 2^exponent. Double-precision uses 1 sign + 11 exponent + 52 significand bits.
  • Roundoff error: The inherent error from approximating real numbers in finite binary fractions. A binary fraction can exactly represent only numbers whose denominator is a power of 2 (1/2, 1/4, 3/8). Most decimal fractions (0.1, 0.2) are infinitely repeating in binary.
  • 0.1 + 0.2 != 0.3: The canonical floating-point surprise. The IEEE 754 approximations of 0.1 and 0.2 sum to a slightly different value than the approximation of 0.3.
  • decimal.Decimal: Python's library for exact decimal arithmetic. Used in financial software. Always pass a STRING argument — passing a float literal already loses precision.
  • Tolerance-based comparison: The safe way to compare floats: abs(x - y) < tolerance (e.g., 1e-9), never x == y.
  • Python int vs. fixed-width int: Python's built-in int is arbitrary-precision and never overflows. Fixed-width types (NumPy int32, struct packing) do overflow.

Worked Examples / Code Walkthroughs

Example 1 — 8-bit unsigned overflow:

Initial value:   200          (binary 11001000)
Add:             100          (binary 01100100)
Naive sum:       300          (binary 1 00101100, 9 bits)
Discard carry:   00101100     (the high bit "1" overflows out)
Stored value:    44

Equivalent: 300 mod 256 = 44.

The register cannot hold a 9-bit value, so it discards the high bit. This is the wraparound behavior.

Example 2 — Floating-point surprise:

print(0.1 + 0.2)            # 0.30000000000000004
print(0.1 + 0.2 == 0.3)     # False

import decimal
print(decimal.Decimal(0.1)) # 0.1000000000000000055511151231257827021181583404541015625
                            # The "true" stored value of the float 0.1 -- not 0.1 exactly

# Fix for currency:
from decimal import Decimal
a = Decimal('0.1')          # STRING -- exactly 0.1
b = Decimal('0.2')
print(a + b)                # 0.3
print(a + b == Decimal('0.3'))  # True

Example 3 — Safe equality with tolerance:

def almost_equal(x, y, tol=1e-9):
    return abs(x - y) < tol

x = 0.1 + 0.2
y = 0.3
print(x == y)            # False (bit-pattern check)
print(almost_equal(x, y))# True (within tolerance)

Example 4 — Currency calculation, the wrong way and the right way:

# WRONG: floating-point currency
price = 19.99
tax_rate = 0.0825
tax = price * tax_rate
print(tax)               # 1.6491750000000001  (NOT 1.65)

# RIGHT: decimal.Decimal
from decimal import Decimal
price = Decimal('19.99')
tax_rate = Decimal('0.0825')
tax = price * tax_rate
print(tax)               # 1.649175  (exact)
print(tax.quantize(Decimal('0.01')))  # 1.65  (rounded to cents)

Common Pitfalls

  • Believing Python integers can overflow. Python's built-in int is arbitrary-precision; it never overflows. Overflow can still happen via NumPy or struct calls into fixed-width types — but pure Python arithmetic does not have an overflow risk.
  • Comparing floats with ==. This will fail unpredictably. Always use a tolerance: abs(x - y) < 1e-9.
  • Passing a float to Decimal. Decimal(0.1) reads the imperfect IEEE 754 approximation; you get a Decimal that is almost 0.1 but not exactly. Always pass a string: Decimal('0.1').
  • Confusing overflow with roundoff. Overflow is a range issue (result too big or too small for the type). Roundoff is a precision issue (real numbers can't be exactly stored). Different causes, different fixes.
  • Trusting that 8-bit unsigned holds 256. The max value is 255, not 256. With 8 bits you have 256 distinct values (0 through 255 inclusive); the max value is exactly 2^n - 1.

Connect to the Exam

Topic 1.4 anchors Big Idea 2 (Data) and produces some of the most rule-rich exam questions in the entire course. Expect: (1) a small arithmetic problem with a fixed-width unsigned or signed integer, asking what value remains after overflow, (2) a code segment using floating-point arithmetic where you must predict the inexact result, (3) a question about why two floats compute to slightly different values mathematically equal, and (4) a real-world risk question (Ariane 5, currency) tying overflow to consequences. The exam tests your ability to apply the formulas: 8-bit unsigned max = 255, 8-bit signed range = -128 to +127, and 0.1 + 0.2 != 0.3 in IEEE 754.

Self-Check

  • An 8-bit unsigned integer holds 250. The program adds 10. What value remains, and what is the modular arithmetic that explains it?
  • Why is 0.5 + 0.5 == 1.0 always True in Python, but 0.1 + 0.2 == 0.3 always False? (Hint: think about which fractions have exact binary representations.)
  • Write a Python function safe_eq(x, y, tol=1e-9) that compares floats safely, and demonstrate that safe_eq(0.1 + 0.2, 0.3) returns True.
  • A C program uses a 32-bit signed int to hold the number of seconds since January 1, 1970. When will this counter overflow? (Hint: 2^31 seconds.)
  • Suppose a banking system has been computing interest using float instead of Decimal for ten years on millions of transactions. Why is this a problem, and how should the bank diagnose how much money has been silently rounded?
