AP CSP — Topic 9.5: Mastering the In-Exam Written Response — Study Streak Tracker Capstone

Core Question

How does a student translate a working program into Written Response prose that maximally captures rubric points across four independent prompt categories — and what patterns in low-scoring responses must be deliberately avoided? The in-exam Written Response is the component of the Create Performance Task that students prepare for by working on their project but are graded on during the exam itself. This section walks through all four prompt categories using the Study Streak Tracker as the running example. Each category has a defined zero-point pattern, a partial-credit pattern, and a full-credit template. Section 9.5 ties together the function-vs-purpose discipline of 9.1, the code-requirements compliance of 9.2, the video evidence of 9.3, and the line-number citation strategy of 9.4 into a coherent in-exam workflow.

Key Concepts

  • Four Prompt Categories: (1) Program Design, Function, and Purpose; (2) Algorithm Development; (3) Errors and Testing; (4) Data and Procedural Abstraction.
  • Independent Scoring: Each category is scored separately. A weak response in one cannot be compensated by strong work in another, but neither does a weak category drag down the others.
  • Category 1 Sub-Items: Purpose (1 pt — audience + problem + benefit), input (1 pt — what the program accepts), output (1 pt — what the program produces).
  • Category 2 Sub-Items: Sequencing (1 pt), selection (1 pt), iteration (1 pt), different-argument consequence (1 pt) — all referenced to the student-developed procedure.
  • Category 3 Sub-Items: Error identified (1 pt — specific, non-trivial bug with root cause), steps taken (1 pt — debugging technique + verification).
  • Category 4 Sub-Items: Data abstraction (1 pt — scale-specific manage-complexity claim), procedural abstraction (1-2 pts — explanation of parameter use and reusability).
  • Three High-Scoring Traits: PPR line-number citation; precise rubric vocabulary (sequencing, selection, iteration); explicit distinction between purpose and function.
  • Trivial-Error Trap: A syntax error fixed in one keystroke is too small for Category 3 credit. The rubric expects a logic or runtime bug requiring real investigation.
  • Specificity Drives Points: Vague abstraction claims fail; concrete scale reasoning (e.g., "100 sessions = 100 variables") earns points.
  • Cohesion Across Categories: Categories build on each other — same audience, same data structures, same procedure. Cohesion signals understanding.
  • Preparation Strategy: Draft all four responses before the deadline; read aloud while looking at the PPR; identify and fix gaps while revision is possible.

PT Rubric Alignment

Section 9.5 maps every Written Response sub-item to a rubric row. Category 1 feeds the Program Purpose and Function row. Category 2 feeds the Algorithm Implementation row and contributes to the Procedural Abstraction row through the different-argument trace. Category 3 is the primary evidence for the Testing row. Category 4 feeds both the Data Abstraction and Procedural Abstraction rows. The Managing Complexity row is touched by Category 4 (data abstraction at scale) and Category 2 (sequencing in the procedure). Six rubric rows; five Written Response categories that supply evidence for them; six total rubric points possible. Mastery of Section 9.5 means knowing which sentence in your response addresses which rubric criterion — and confirming that no sub-item is left unaddressed.

Common Pitfalls

  • Answering the wrong question in Category 1. Writing function ("tracks study sessions") instead of purpose (audience + problem + benefit) is the textbook zero-point response. Always name the human audience and the specific problem they face. (See Topic 9.1.)
  • Vague algorithm description in Category 2. "It iterates and checks conditions" earns nothing. The rubric requires specific construct names tied to specific lines: "iteration at line 35, selection at line 37, sequencing across phases A-D." Use the rubric's vocabulary, not synonyms.
  • Trivial error in Category 3. A syntax-error story earns zero. Pick a real logic or runtime bug whose root cause was not immediately obvious. The Study Streak Tracker timedelta-vs-int bug is the canonical example: the comparison failed silently because Python returned a timedelta where an int was expected.
  • Vague abstraction claim in Category 4. "My list manages complexity" or "my procedure abstracts well" earns nothing. Quantify the scale problem (100 sessions = 100 variables) and name the specific reusability mechanism (different target_subject argument changes the filter).
  • No PPR citation. A Written Response that says "my procedure does X" without citing PPR Screenshot 2 lines makes the reader trust the writer. A response that says "as shown in Screenshot 2 lines 28-73, my procedure does X" makes the claim falsifiable. Always cite.

Connect to the Exam

Section 9.5 is the integration point for the entire Create Performance Task. By exam day, your code is fixed (submitted earlier) and your video is fixed (submitted earlier). The Written Response is the only artifact you compose under exam timer pressure, with only your PPR as a reference. Practiced templates make this feasible. For each category, know roughly which sentences you will write before you sit down. For Category 1, draft purpose + input + output sentences keyed to your specific audience. For Category 2, draft a paragraph that names sequencing, selection, and iteration with PPR line numbers and traces what changes with a different argument. For Category 3, prepare a real bug story with root cause and verification step. For Category 4, draft scale-specific data abstraction reasoning and parameter-driven procedural abstraction reasoning. The students who walk in with prepared substance for each category, then adapt to the exact prompt wording, capture the most points.

Self-Check

  • Write a draft Category 1 response for the Study Streak Tracker. Verify it names a specific audience, identifies a concrete problem, articulates a tangible benefit, AND identifies the inputs and outputs of the program. Compare against the 0-point and full-credit examples in Section 9.5.
  • Write a draft Category 2 response that references the procedure compute_streak(records, target_subject). Cite specific line numbers from your PPR (you may use placeholder line numbers if your PPR is not yet complete). Identify sequencing across at least two distinct phases inside the procedure body. Trace what changes if you call the procedure with target_subject = "english" instead of target_subject = "math".
  • Reflect on a real bug you encountered while writing your Create PT program (or, if none, study the Study Streak Tracker timedelta bug from Section 9.5). Write a Category 3 response that names the bug, explains the root cause, describes the debugging technique you applied, and describes a verification test. Verify your response would not earn the trivial-error 0 because it is not a one-keystroke fix.
  • Write a Category 4 response that solves the scale-specificity problem. Quantify the manage-complexity claim ("if the student logs 100 sessions...") and name the specific reusability mechanism for your procedure. Verify the response distinguishes data abstraction from procedural abstraction with concrete examples for each.
  • Read all four of your draft responses aloud while looking at your PPR. Do the line numbers you cite actually exist in Screenshot 2? Do the variable names you reference appear in the screenshots? Identify any gap between Written Response and PPR — that gap is fixable while revision is still possible.
