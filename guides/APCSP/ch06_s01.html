AP CSP — Topic 6.1: Defining Procedures

Core Question

What is a procedure, why does the Create Performance Task require one, and how do you write a parameterized procedure that satisfies the rubric in both Python and AP Pseudocode?

Key Concepts

  • A procedure is a named, packaged block of code that can be called by name. Defining a procedure once and calling it many times replaces error-prone copy-paste.
  • In Python, a procedure starts with def name(parameters): and uses indentation to delimit the body. A value-returning procedure ends with return.
  • In AP Pseudocode, a procedure starts with PROCEDURE name(parameters) and uses curly braces to delimit the body. The return statement is RETURN, and assignment uses <- instead of =.
  • A void procedure performs an action but does not hand back a useful value; in Python it implicitly returns None. A value-returning procedure computes a result and sends it back through return.
  • Parameters are placeholder names listed in the header. Arguments are the actual values supplied at the call site.
  • The Create PT rubric requires a student-developed parameterized procedure whose body contains sequencing, selection, AND iteration. All three constructs must live inside the body, not in the surrounding script.
  • Tracker v1 hardcoded the string 'math' inside its loop, so counting a second subject required duplicating the loop. Tracker v2 makes the subject a parameter, so one procedure works for any subject.
  • Procedural abstraction is the broader practice of replacing repeated or complex logic with a named procedure. It is the foundation of the modular design Margaret Hamilton pioneered for the Apollo Guidance Computer.
  • A procedure call temporarily binds each argument to the matching parameter, runs the body, and (for value-returning procedures) sends a result back via return.
  • Procedures should be written so that the same arguments always produce the same return value. This makes them easy to test in isolation.

Worked Examples / Code Walkthroughs

Example 1 — A minimal value-returning procedure in Python

def greet(name):
    message = "Hello, " + name + "!"
    return message

result = greet("Alex")     # 'Alex' is the argument bound to parameter name
print(result)              # Output: Hello, Alex!

The line def greet(name): introduces the procedure. Inside, name is a parameter — a placeholder that has no value until the call happens. When greet("Alex") runs, name is set to "Alex" for the duration of the call, the body builds the message, and return message hands the value back to be stored in result.

Example 2 — Same procedure in AP Pseudocode

PROCEDURE greet(name)
{
    msg <- "Hello, " + name
    RETURN msg
}

result <- greet("Alex")

Notice the four mechanical translations: def becomes PROCEDURE, the colon and indentation become curly braces, = becomes <-, and lowercase return becomes uppercase RETURN.

Example 3 — Tracker v2 procedure shell (the rubric pattern)

def compute_streak(records, target_subject):
    qualifying_dates = []
    for record in records:                      # iteration
        d, subject, minutes = record
        if subject == target_subject and minutes >= 30:   # selection
            qualifying_dates.append(d)          # sequencing
    return len(qualifying_dates)                # placeholder return; full version computes streak

This shell already satisfies the Create PT rubric for "sequencing + selection + iteration inside the procedure." The full body in 6.5 adds the streak computation, but the rubric structure is already in place.

Common Pitfalls

  • Putting the loops and ifs in the main script and only the return inside the procedure. The rubric requires the constructs to live inside the body, so this loses the point.
  • Confusing parameters with arguments. The exam rewards using each term precisely: parameters are in the header, arguments are at the call site.
  • Forgetting the colon at the end of a Python def line, or forgetting to indent the body. Both are syntax errors.
  • Writing a void procedure when a value-returning procedure is required, then trying to use the (None) return value in an expression.
  • Mixing notations on the AP exam — for example, using def with curly braces, or using RETURN in Python. Each notation has its own keywords and you must stay consistent.

Connect to the Exam

The Create PT (worth six rubric points) explicitly tests whether your submitted program contains a student-developed parameterized procedure whose body has sequencing, selection, and iteration. This single section sets up the most heavily weighted code requirement of the entire course. The MCQ portion of the AP exam also routinely shows a procedure header in either notation and asks you to translate, predict the return value, or distinguish parameters from arguments — concepts you must lock in here before facing 6.2's deeper data-flow questions.

Self-Check

  1. Translate this Python header to AP Pseudocode: def square(n):.
  2. In one sentence, what is the difference between a void procedure and a value-returning procedure?
  3. A classmate writes a 30-line program where the only procedure is def show(): return 'done' (with all the loops and ifs in the main script). Does this satisfy the Create PT rubric? Why or why not?
  4. After result = greet("Alex") runs, can you reference the variable name in the main script? Why?
  5. What is the difference between a parameter and an argument in compute_streak(study_records, "math")?
