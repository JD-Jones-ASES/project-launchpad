AP CSP — Topic 2.2: Variables, Assignment, and Memory

Core Question

How does a program remember things, and what is actually happening when you write score = 42? A program that cannot remember anything cannot do anything useful. Variables are Python's mechanism for naming locations in memory so a value can be stored, retrieved, updated, and passed around. This section unpacks the mechanics of assignment in Python, contrasts Python's = with AP Pseudocode's <-, introduces the integer and string types and their + behavior, and establishes the snake_case naming convention that makes code readable to other humans.

Key Concepts

  • Variable: A named storage location in memory holding a single value at any given time. Created the first time a name appears on the left side of an assignment statement. Python variables have no fixed type — the same name can hold an integer, then a string, then a list.
  • What score = 42 does: (1) Python evaluates the right-hand side; (2) allocates (or updates) memory; (3) binds the name score to that memory region. Future reads of score retrieve the bound value.
  • Assignment vs equality: In Python, = assigns and == tests equality. In AP Pseudocode, <- assigns and a single = (inside a condition like IF (x = 10)) tests equality. Confusing the two is a recurring point-loss error.
  • snake_case: Python's variable-naming convention — all lowercase, underscores between words (student_name, total_score). Beyond the syntactic rules (no leading digit, no keywords), good names are descriptive (student_age vs x) and avoid Python built-ins like list, input, print, sum.
  • int: Whole numbers, positive or negative, no decimal. Python integers grow as needed — no fixed upper limit.
  • str: A sequence of characters in quotes. "42" is a string, not a number, even though it consists only of digits.
  • + overloading: Integer addition on ints (17 + 3 = 20), concatenation on strings ("17" + "3" = "173"). Mixing int and string with + raises a TypeError.
  • Variable swap idiom: The naive a = b; b = a fails because the first line overwrites a. Either use a temp variable or use Python's a, b = b, a tuple assignment, which evaluates all right-hand values before any assignment.
  • Mutability: Strings and ints are immutable; lists are mutable. name[0] = "B" on a string raises a TypeError; on a list it works fine. (Lists are covered fully in Unit 5.)

Worked Examples / Code Walkthroughs

1. Trace what happens when a variable is updated.

score = 42        # bind name 'score' to value 42
score = score + 5 # evaluate (42 + 5)=47, then bind 'score' to 47
print(score)      # output: 47

The expression score + 5 is evaluated first — Python looks up the current value of score (42), adds 5, and stores 47 back into score. The old 42 is gone. This sequencing is why x = x + 1 is not a mathematical equality claim but a perfectly normal Python statement.

2. Python and AP Pseudocode side by side. The same swap of values written in both notations:

# Python
total = 0
total = total + 10

AP Pseudocode equivalent:

total <- 0
total <- total + 10

The pseudocode arrow <- makes the direction of data flow visible. AP graders will accept either notation in free-response if you choose, but the pseudocode arrow is what appears on the multiple-choice exam.

3. The string-vs-integer pitfall.

age_int = 17
age_str = "17"
print(age_int + 3)     # 20 (arithmetic)
print(age_str + "3")   # "173" (concatenation)
print(type(age_int))   # <class 'int'>
print(type(age_str))   # <class 'str'>

Reading input() (covered in Section 2.4) always returns a string, so this trap is the #1 cause of TypeError: can only concatenate str (not "int") to str in beginner Python.

4. The Python tuple-assignment swap.

a = 10
b = 20
a, b = b, a   # right-hand evaluated first: (20, 10)
print(a, b)   # 20 10

Python reads both right-hand-side values into a tuple before doing any assignment, so neither original value is destroyed before being read. This is the idiomatic Python swap.

Common Pitfalls

  • Confusing = with == inside an if. if x = 5: is a syntax error. The fix is if x == 5:.
  • Mixing AP Pseudocode and Python notations. Use <- only in pseudocode answers; use = only in Python. Do not mix on free-response items.
  • Treating "42" as a number. "42" + 3 raises a TypeError. Convert with int("42") first.
  • Naive swap. a = b; b = a does not swap — it copies b into both. Use a temp variable or a, b = b, a.
  • Shadowing built-ins. Writing list = 5 silently overwrites Python's list constructor; later code that calls list(...) crashes confusingly with "int object is not callable."

Connect to the Exam

This section maps to Big Idea 1 (Creative Development) and Big Idea 3 (Algorithms and Programming). Expect multiple-choice questions that ask you to (a) trace a sequence of assignments and predict the final variable value, (b) translate between Python = and AP Pseudocode <-, and (c) predict the output of an expression mixing integer and string operations. The most-tested trap is the + operator's two meanings: arithmetic on numbers, concatenation on strings. If a multiple-choice item shows "5" + 3, the answer is almost always TypeError; if it shows "5" * 3, the answer is "555".

Self-Check

  • Translate the following Python statements into AP Pseudocode: total = 0 and total = total + 10. Where does the arrow appear, and why is the visual distinction useful?
  • Predict the output of print("5" * 3) and explain in one sentence why the answer is not 15.
  • A student writes if score = 100: and gets a SyntaxError. Rewrite the line correctly and name the error category from Section 2.5.
  • Explain in your own words why a = b; b = a does NOT swap two variables, and contrast it with how Python's a, b = b, a succeeds.
  • Identify which name follows snake_case: studentName, student_name, STUDENT-NAME, s_n. Then say which of the four is the most descriptive.
