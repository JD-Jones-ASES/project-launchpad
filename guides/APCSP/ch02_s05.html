AP CSP — Topic 2.5: Errors and Documentation

Core Question

How do you classify a programming error so you know how to find and fix it, and how do well-written comments prevent bugs before they happen? Every programmer writes buggy code. The difference between a struggling programmer and a productive one is not the absence of bugs but the speed of diagnosis — which depends on knowing what kind of bug you have. Python distinguishes three fundamentally different error types, each requiring a different approach. This section establishes the vocabulary, then connects it to comment style: comments that explain why keep human understanding aligned with what the code actually does, preventing logic errors before they appear.

Key Concepts

  • Three error categories: Syntax (the code cannot be parsed; Python refuses to run it), Runtime (the code starts but crashes at an illegal operation), Logic (the code runs to completion but produces wrong output). Syntax and runtime errors announce themselves; logic errors are silent.
  • Syntax error: A violation of Python's grammar rules detected by the parser before any code runs. Common causes: missing colon after if/for/def, unmatched parenthesis or quotation mark, misspelled keyword, incorrect indentation.
  • Runtime error: A crash during execution. Python prints a traceback identifying the file, line, and error type. Common types: ZeroDivisionError (10 / 0), TypeError ("hello" + 5), ValueError (int("twenty")), NameError (using an unassigned name), IndexError (accessing past the end of a list — covered in Unit 5).
  • Logic error: The most dangerous category. The program runs to completion without crashing but produces wrong output. No error message, no traceback — just a wrong answer. Detected only through testing with known-correct outputs.
  • TypeError vs ValueError: TypeError = wrong type entirely ("hello" + 5). ValueError = right type but wrong content (int("hello") — string is the right argument type for int(), but the content cannot be parsed as a number).
  • Reading a traceback: Bottom-up. The last line names the error type and message; the lines above identify the file and line in your code. Skilled programmers read the entire traceback because bugs usually live in code they wrote, not in Python's libraries.
  • Comments explain WHY, not WHAT: Anything after # on a line is ignored by Python. Good comments explain decisions a future reader cannot infer from the code itself. Poor comments restate what the code already says.
  • PPR comment rule: For the Create PT, no comments are allowed in the PPR screenshots of the list initialization/use and the parameterized procedure. Comments in those regions disqualify the submission from the abstraction points. Comment freely elsewhere; capture comment-free regions for the PPR.

Worked Examples / Code Walkthroughs

1. Syntax error — missing colon.

x = 10
if x > 5
    print("x is greater than 5")

Python's response:

File "example.py", line 2
    if x > 5
            ^
SyntaxError: expected ':'

The caret (^) points at the column where parsing failed. Python's syntax messages are extremely helpful — they pinpoint the line and often the character.

2. Three runtime errors side by side.

# ZeroDivisionError
result = 10 / 0                       # ZeroDivisionError: division by zero

# TypeError (str + int)
age = input("Age? ")                  # returns "17"
total = age + 5                       # TypeError: can only concatenate str (not "int") to str

# ValueError (string is right TYPE for int(), but wrong CONTENT)
count = int("twenty")                 # ValueError: invalid literal for int() with base 10: 'twenty'

Each prints a traceback identifying the offending line and the error type. ZeroDivisionError signals undefined math; TypeError signals incompatible operands; ValueError signals unparseable content.

3. Logic error — silent and dangerous.

score1, score2, score3 = 80, 90, 70

# BUGGY: precedence error
average = score1 + score2 + score3 / 3
# Python evaluates as: 80 + 90 + (70/3) = 80 + 90 + 23.33 = 193.33
print("Average:", average)            # 193.33 — WRONG, no error message

# CORRECT: parentheses enforce intended order
average = (score1 + score2 + score3) / 3
print("Average:", average)            # 80.0 — correct

The buggy line runs without crashing and prints a plausible number. Only careful testing — comparing the printed average against a known-correct value — reveals the defect. This is exactly why logic errors are the most dangerous category.

4. Effective vs ineffective comments.

# Poor — restates what the code already shows
x = 10              # set x to 10
y = x * 2           # multiply x by 2

# Effective — explains WHY (decisions a reader cannot infer)
CELSIUS_TO_FAHRENHEIT_OFFSET = 32   # required by the formula F = (C * 9/5) + 32

def celsius_to_fahrenheit(celsius):
    # Multiply first to avoid integer truncation in the fraction 9/5
    return (celsius * 9 / 5) + CELSIUS_TO_FAHRENHEIT_OFFSET

The constant CELSIUS_TO_FAHRENHEIT_OFFSET uses UPPER_SNAKE_CASE, Python's convention for values that should not change during execution. The comment explains why the constant exists, not just what its value is.

Common Pitfalls

  • Mistaking a logic error for a "fluke." If the program runs without error but produces wrong output, you have a logic error — investigate, do not retry.
  • Reading only the last line of a traceback. The last line identifies the error type, but the line above shows where in your code it happened. Read bottom-up but read the whole thing.
  • Confusing TypeError with ValueError. TypeError = wrong type ("hello" + 5). ValueError = right type, wrong content (int("hello")).
  • Thinking syntax errors live in your IDE. Syntax errors are real Python errors with line numbers. Trust the message; the line and column it points at are correct.
  • Including comments in PPR screenshots. Automatic disqualification from the abstraction points. Comment-strip the regions you screenshot for the Create PT.

Self-Check

  • Classify each error as syntax, runtime, or logic, and explain your reasoning: (a) print("Hello (missing closing quotation mark); (b) x = 5 / 0; (c) a sales-tax program that uses 0.008 instead of 0.08.
  • Rewrite this poor comment as an effective one: # multiply price by rate before the line tax = price * rate. Choose a why a programmer might want to leave for a future reader.
  • A program crashes with TypeError: can only concatenate str (not "int") to str. Name the most likely cause from Unit 2 and write the fix.
