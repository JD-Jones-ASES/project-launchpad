AP CSP — Topic 4.3: Algorithm Tracing

Core Question

Once you have written a loop, how do you verify it actually does what you intended? Running the program and looking at output is not enough — you cannot tell from a single output whether the loop ran the right number of iterations, processed the right values, or initialized variables correctly. Algorithm tracing is the discipline of walking through code line by line, by hand, recording every variable's value after every step. It is how professional programmers check correctness, and it is the exact skill the AP exam tests when it asks "What does this code print?" Mastering Topic 4.3 means building trace tables fluently and using them to detect both off-by-one errors and logic errors.

Key Concepts

  • Algorithm tracing: Walking through code line by line, by hand, simulating the computer's execution to record variable values after each step.
  • Trace table: A hand-drawn table with one column per variable (plus iteration counter and output column) and one row per iteration. The standard tool for visualizing loop execution.
  • Off-by-one error (too few): Loop runs one fewer iteration than intended. In Python: range(N) when you wanted range(1, N+1), or < when you wanted <=.
  • Off-by-one error (too many): Loop runs one more iteration than intended. In Python: range(1, N+1) when you wanted range(1, N), or <= when you wanted <.
  • Logic error: The loop runs the right number of times but performs the wrong operation, computing a plausible-looking but incorrect result.
  • Initialization error: Subset of logic errors. Setting total = 1 instead of total = 0 corrupts a sum; setting product = 0 instead of product = 1 zeroes a product.
  • Operator error: Using * when you meant +, or > when you meant <. The loop terminates correctly but the math is wrong.
  • Why tracing finds bugs: Comparing actual variable values to expected values exposes the exact iteration where the algorithm diverges from its intended behavior.
  • Tracing for loops: One row per value of the loop variable. range(1, 5) → four rows (i = 1, 2, 3, 4). The loop exits when the range is exhausted.
  • Tracing while loops: Record the condition's truth value at the start of each iteration. The loop exits when the condition is false at the top of the next would-be iteration.

Worked Examples / Code Walkthroughs

Example 1 — Tracing a correct sum

N = 4
total = 0
for i in range(1, N + 1):
    total = total + i
print(total)

Trace table:

iteration i total
start - 0
1 1 1
2 2 3
3 3 6
4 4 10

After iteration 4, range is exhausted. print(total) → 10. Confirmed: 1+2+3+4 = 10.

Example 2 — Spotting an off-by-one error via tracing

N = 5
total = 0
for i in range(N):    # BUG: starts at 0, not 1
    total = total + i
print(total)

Trace:

iteration i total
start - 0
1 0 0
2 1 1
3 2 3
4 3 6
5 4 10

print(total) → 10, not the intended 15. The trace makes the bug obvious: i started at 0 (not 1) and never reached 5. Fix: range(1, N+1).

Example 3 — Spotting a logic error via tracing

N = 4
total = 1                      # BUG 1: should be 0 for a sum
for i in range(1, N + 1):
    total = total * i          # BUG 2: should be + not *
print(total)

Trace:

iteration i total
start - 1
1 1 1
2 2 2
3 3 6
4 4 24

print(total) → 24. This is 4! (factorial), not the intended sum 10. Two bugs compound: the wrong initialization (1 acts as the multiplicative identity) and the wrong operator (* instead of +). Without tracing, the output 24 looks plausible. Tracing instantly reveals the algorithm is computing the wrong function.

Example 4 — Tracing a while loop

x = 1
result = 0
while x <= 4:
    result = result + x
    x = x + 1
print(result)

Trace:

iteration x at top x <= 4? result x after
start 1 true 0 1
1 1 true 1 2
2 2 true 3 3
3 3 true 6 4
4 4 true 10 5
exit 5 false 10 5

print(result) → 10. The condition becomes false when x = 5, exiting the loop. Final state: x = 5, result = 10.

Common Pitfalls

  • The off-by-one error (the headliner): Running the loop one too few or one too many times. Tracing the first and last iterations explicitly catches this — set up the table, count rows, compare to expected count.
  • Skipping the trace and "estimating" the answer: AP exam tracing questions do not yield to estimation. Always set up an actual trace table on scratch paper and fill it in.
  • Logic errors disguised by plausible-looking output: 24 looks like a reasonable answer for a sum question. Verifying via trace catches mismatches between expected and actual variable progressions.
  • Forgetting to record output rows: If the algorithm produces output, your trace table must have an output column. Otherwise you cannot answer "What does this print?"
  • Skipping the initial row: The "start" row (variables before the first iteration) is essential. Many bugs (especially initialization errors) appear there.

Connect to the Exam

Tracing is the core technique behind every AP exam algorithmic-analysis question. When you see "What does the following code print?" or "What is the final value of variable X?", you are being asked to trace the loop. Off-by-one errors are the single largest source of point loss on these questions — set up a trace table and explicitly check whether the loop's first and last iterations match expectation. Logic errors are tested through "Identify the bug" questions, where the expected and actual outputs differ in a subtle way. Always trace at least the first two and last two iterations of any loop you are analyzing on the exam; estimating leads to mistakes.

Self-Check

  • Construct a trace table for the following code and state what it prints. x = 10; y = 0; while x > 0: y = y + x; x = x - 3; print(y).
  • The loop for i in range(1, 6): print(i) is supposed to print five values. Verify with a trace table, listing every value of i.
  • A student writes total = 1; for i in range(1, 5): total = total + i; print(total). Trace it and identify the bug.
  • A student writes for i in range(0, 6): total = total + i to compute the sum 1+2+...+5. Trace this and identify the off-by-one error.
  • Explain in your own words why tracing is the only reliable way to distinguish a logic error from a correct algorithm, even when both produce a plausible-looking output.
